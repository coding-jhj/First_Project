# 팀원별 예상 Q&A 통합본 (전체 5명)

## 문수찬 (Voice / Q&A)

### 1) STT가 잘못 인식하면 어떻게 되나요?
- 현재 설계는 미인식/오인식 시 `unknown`으로 분류하고 기본 안전 안내 흐름으로 복귀합니다.
- 기능이 멈추는 대신 장애물 안내를 계속 제공하도록 안정성을 우선했습니다.

### 2) STT는 오프라인인가요?
- 서버 `src/voice/stt.py`는 Google STT를 사용해 네트워크가 필요합니다.
- Android 경로는 내장 SpeechRecognizer를 사용하므로 앱 단독 운용이 가능합니다.

### 3) TTS 캐시는 왜 필요한가요?
- 동일 문장을 매번 합성하면 지연이 누적됩니다.
- 해시 기반 캐시로 반복 경고 문장을 즉시 재생하도록 최적화했습니다.

### 4) 위험 상황에서 음성 우선순위는 어떻게 동작하나요?
- `critical / beep / silent` 계층을 사용합니다.
- 위험도에 따라 즉시 음성, 비프, 무음(UI만)으로 분기해 피로도를 줄였습니다.

### 5) 음성이 겹쳐서 들리는 문제는 어떻게 대응했나요?
- 서버는 빠른 캐시 반환으로 중첩 가능성을 줄였습니다.
- 앱에서는 최신 위험 문장을 우선 재생하도록 재생 제어 정책을 적용합니다.

### 6) TTS 품질을 어떻게 관리하나요?
- 한국어 음성(`ko-KR-SoonBokNeural`)을 고정 사용합니다.
- 일반 안내와 긴급 안내의 말하기 속도를 분리해 전달력을 높였습니다.

### 7) 특수문자나 긴 문장에서 TTS가 깨질 수 있지 않나요?
- SSML 삽입 전에 텍스트를 이스케이프해 문법 오류를 방지했습니다.
- API 응답 문구가 바뀌어도 합성 안정성을 유지합니다.

### 8) 서버 시작 직후 첫 TTS가 느린 이유와 해결은?
- 클라우드 음성 합성은 첫 호출에서 초기 지연이 발생할 수 있습니다.
- 서버 시작 시 `warmup_cache()`로 샘플 문장을 미리 생성해 지연을 완화했습니다.

### 9) 데모 직전 어떤 음성 테스트를 해야 하나요?
- `/stt` 호출로 텍스트/모드 반환 확인,
- `/tts` 호출로 wav 파일 반환 확인,
- 앱에서 짧은 경고 문장과 긴 설명 문장 각각 1회 재생 확인합니다.

### 10) 향후 개선 계획은 무엇인가요?
- STT 동의어 확장 및 발화 정규화 고도화,
- 환경 소음별 임계값 튜닝,
- 음성 큐 우선순위(긴급 문장 선점) 정책을 더 세밀하게 개선할 계획입니다.

## 정환주 (팀장 / Server / Frontend)

### 1) MVP 범위를 어떻게 잘랐나요?
- 발표 필수인 탐지-문장-음성 파이프라인과 `/detect`, 대시보드, 문서화를 핵심으로 고정했습니다.
- 부가 기능은 데모 안정성 기준으로 후순위 처리했습니다.

### 2) 서버 통합에서 가장 어려웠던 점은?
- 비전/깊이/NLG 모듈이 서로 다른 출력 포맷을 가지는 점이었습니다.
- `routes.py`에서 공통 응답 스키마로 정규화해 해결했습니다.

### 3) `/detect` API 설계 원칙은 무엇인가요?
- 실패 상황에서도 음성 안내가 끊기지 않도록 항상 `sentence`를 반환하도록 설계했습니다.

### 4) DB와 tracker를 왜 분리했나요?
- tracker는 프레임 단위 단기 안정화(EMA), DB는 공간 기억 같은 상태 저장(중장기) 역할이라 책임 분리가 필요했습니다.

### 5) Cloud Run 배포 선택 이유는?
- 빠른 배포/롤백, 팀 협업 편의, 운영 부담 감소 때문이며 MVP 기간에 적합했습니다.

### 6) 대시보드의 목적은 뭔가요?
- 모델 상세 디버깅보다 현재 인식/경고/응답 상태를 발표 중 빠르게 확인하는 운영 가시성에 초점을 맞췄습니다.

### 7) 서버 장애 시 앱은 어떻게 대응하나요?
- 온디바이스 경로를 유지해 핵심 안내가 멈추지 않도록 이중 경로로 설계했습니다.

### 8) 성능 저하를 어떻게 관리했나요?
- 이미지 처리 흐름 단순화, 워밍업, 캐시, 불필요한 연산 제거로 지연을 통제했습니다.

### 9) README/문서 정리 기준은?
- 설치-실행-데모 시나리오-트러블슈팅 순서로, 비개발자도 따라할 수 있게 구성했습니다.

### 10) 최종 검수에서 확인한 핵심 체크는?
- 서버 기동, `/detect` 정상 응답, 대시보드 반영, 음성 출력, 데모 시나리오 재현성 5가지를 반복 확인했습니다.

## 신유득 (Vision / ML)

### 1) YOLO 모델 선택 기준은?
- 실시간성과 정확도 균형을 기준으로 선택했습니다.
- 실제 보행 안전에서 오탐/미탐 리스크가 낮은 설정을 우선했습니다.

### 2) 계단 클래스는 왜 별도 관리했나요?
- 일반 객체보다 안전 리스크가 커서 별도 학습/평가 포인트로 관리했습니다.

### 3) Depth와 YOLO를 같이 쓰는 이유는?
- YOLO는 객체 인식에, Depth는 거리/바닥 구조 판단에 강해 상호 보완됩니다.

### 4) benchmark는 어떤 지표를 봤나요?
- 추론 시간, 탐지 안정성, 위험 상황 재현성, 실패 케이스 비율을 핵심으로 측정했습니다.

### 5) 실패 케이스는 주로 어디서 발생했나요?
- 역광/저조도, 가림(occlusion), 빠른 이동체, 작은 물체에서 성능 저하가 컸습니다.

### 6) 성능 자료 정리는 어떻게 했나요?
- 모델 파일명/버전/설정값과 결과를 매핑해 재현 가능하도록 기록했습니다.

### 7) threshold 튜닝 원칙은?
- 안전 우선 시나리오에서 미탐을 줄이는 방향으로 조정했습니다.
- 오탐은 후속 문장 로직에서 완화했습니다.

### 8) 추론 지연이 길어지면 어떻게 하나요?
- 입력 해상도/주기 조절, 모델 경량화, 불필요 후처리 제거로 단계적으로 대응합니다.

### 9) 실제 환경 일반화는 어떻게 확인했나요?
- 실내/실외, 조명/거리 변화 조건에서 반복 샘플링해 편향 여부를 확인했습니다.

### 10) 다음 개선 우선순위는?
- 실패 케이스 중심 데이터 보강, 경계 상황 정밀도 향상, benchmark 자동화 강화입니다.

## 김재현 (Android / UX)

### 1) CameraX를 선택한 이유는?
- Android 공식 스택이라 기기 호환성과 생명주기 관리 안정성이 높습니다.

### 2) ONNX 온디바이스 장점은?
- 네트워크 없이도 즉시 추론 가능해 이동 중 안정성이 높습니다.

### 3) 권한 처리에서 중점은?
- 카메라/마이크 권한 실패 시 대체 안내를 명확히 주고 앱이 멈추지 않게 했습니다.

### 4) 발열 이슈는 어떻게 완화했나요?
- 캡처 주기 조절, 불필요 연산 최소화, 백그라운드 상태 처리로 지속 사용 온도를 관리했습니다.

### 5) TTS 겹침은 어떻게 제어하나요?
- 긴급 문장 우선, 중복 문장 스킵, 재생 중단/갱신 정책으로 음성 충돌을 줄였습니다.

### 6) UX 개선에서 가장 큰 변화는?
- 첫 화면과 버튼 흐름을 단순화해 사용자가 바로 행동할 수 있게 한 점입니다.

### 7) 서버 모드와 온디바이스 모드는 어떻게 전환되나요?
- 서버 연결 가능 여부로 분기하고, 실패 시 자동으로 온디바이스로 폴백합니다.

### 8) 접근성 관점에서 반영한 점은?
- 시각 의존도를 낮추기 위해 음성 중심 피드백과 큰 상호작용 단위를 우선 적용했습니다.

### 9) APK 안정성은 어떻게 검증했나요?
- 권한-실행-재실행-백그라운드 전환-복귀 시나리오를 반복 점검해 크래시를 확인했습니다.

### 10) 발표 직전 체크포인트는?
- 권한 허용 상태, 카메라 프리뷰, 음성 출력, 모드 전환, 발열 상태 5가지를 확인합니다.

## 임명광 (NLG / 서버 보조)

### 1) NLG의 핵심 목표는 무엇인가요?
- 정확한 정보 전달, 즉시 행동 유도, 한국어 자연스러움의 균형입니다.

### 2) alert mode는 왜 필요한가요?
- 모든 상황을 같은 강도로 읽으면 피로도가 커져 위험도 기반 계층화가 필요합니다.

### 3) 문장 템플릿 방식 장점은?
- 응답 일관성, 디버깅 용이성, 발표 데모 예측 가능성이 높습니다.

### 4) 한국어 조사 처리는 왜 중요하죠?
- 조사 오류는 신뢰도를 즉시 떨어뜨리므로 자동 조사 처리로 품질을 보장했습니다.

### 5) 너무 긴 문장 문제는 어떻게 막았나요?
- 위험 우선 정보만 남기고 부가 정보를 후순위로 보내 문장을 짧게 유지했습니다.

### 6) 서버 응답 문구는 어떻게 검수했나요?
- 모드별 대표 시나리오를 구성해 부자연/중복/모호 표현을 반복 수정했습니다.

### 7) 동일 상황 반복 안내는 어떻게 처리하나요?
- 변화 감지 기반으로 새 정보 중심 문장을 만들고 반복성 문구는 줄였습니다.

### 8) 오탐이 있을 때 문장 전략은?
- 단정 표현을 줄이고 안전 행동 중심 문구를 우선해 오안내 리스크를 낮춥니다.

### 9) 테스트는 어떻게 했나요?
- 거리/방향/위험도 조합별 출력 케이스를 만들어 회귀 검증했습니다.

### 10) 다음 개선 과제는?
- 사용자 발화 톤 맞춤형 문장 스타일, 상황별 문장 길이 자동 조절, 오류 복구 문구 고도화입니다.
