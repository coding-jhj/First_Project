# 핸드폰·책·마우스 등 탐지 안 되는 원인 분석

## 결론부터

여러 원인이 겹쳐 있으며, **가장 큰 원인은 confThreshold(0.40)** 입니다.  
코드 버그가 아니라 모델과 설정의 한계입니다.

---

## 원인 1 — confThreshold 0.40이 너무 높다 (가장 유력)

```
TfliteYoloDetector.kt 25번째 줄
private val confThreshold = 0.40f
```

이 값 미만의 탐지는 **전부 버려집니다.**

yolo11n은 nano(초경량) 모델입니다. COCO 전체 벤치마크에서 mAP가 약 39% 수준으로,  
**작은 물체나 일부 클래스는 구조적으로 0.40 이하로 나올 수밖에 없습니다.**

| 물체 종류 | 왜 신뢰도가 낮게 나오는가 |
|---|---|
| 휴대폰 | 화면에 작게 보임, 각도·반사에 민감 |
| 책 | 표지 종류가 너무 다양, 세워두거나 겹쳐있으면 더 어려움 |
| 마우스 | 크기가 매우 작음, 배경과 구분 어려움 |
| 리모컨 | 비슷한 형태가 많아 혼동 |

→ 0.25~0.30으로 낮추면 탐지될 가능성이 높습니다.  
  단, 오탐지(없는 걸 있다고 함)도 같이 늘어납니다.

---

## 원인 2 — 320×320 해상도 한계

카메라 원본(예: 1080×1920)이 320×320으로 줄어드는 과정에서  
**마우스 같은 작은 물체는 20×20픽셀 이하**로 쪼그라들 수 있습니다.

nano 모델은 이렇게 작은 물체를 잘 탐지하지 못합니다.

```
원본 마우스 크기: 약 150×100px (1080px 기준)
320으로 줄면:  150 × (320/1080) ≈ 44px
```

44px짜리 물체를 nano 모델로 잡는 건 어렵습니다.

---

## 원인 3 — 노트북(63번)이 COCO_KO에서 빠져 있음

```
VoiceGuideConstants.kt 4번째 줄 주석:
// 제외: 63(노트북 — 이미 포함) ← 주석은 포함이라고 하지만

실제 맵:
62 to "티비"
// 63 없음 ← 실제로 빠져 있음
64 to "마우스"
```

노트북(class ID 63)은 모델이 탐지해도 코드에서 `COCO_KO[63] = null → continue`로 버려집니다.  
주석이 잘못 적혀 있어서 생긴 누락입니다.

---

## 원인 4 — take(8) 상위 8개 제한

```
TfliteYoloDetector.kt
return nms(candidates.sortedByDescending { it.confidence }).take(8)
```

신뢰도 높은 순서로 최대 8개만 반환합니다.  
만약 사람·자동차 등이 이미 8개를 채우고 있으면  
신뢰도가 조금 낮은 책·마우스는 목록에서 잘립니다.

---

## 원인 5 — 파인튜닝 데이터 편향 가능성

강사님이 어떤 데이터로 파인튜닝하셨는지 알 수 없습니다.  
만약 실외 보행 위주(사람, 자동차, 신호등 등)로 학습했다면  
실내 소형 물체(마우스, 책, 리모컨)는 기본 COCO 성능보다 낮을 수 있습니다.

→ 강사님께 확인 필요: "어떤 데이터로 파인튜닝 하셨나요?"

---

## 정리

| 원인 | 해결 방법 | 난이도 |
|---|---|---|
| confThreshold 너무 높음 | 0.40 → 0.25~0.30으로 낮추기 | 쉬움 (숫자 하나) |
| 노트북(63번) 누락 | COCO_KO에 `63 to "노트북"` 추가 | 쉬움 (한 줄 추가) |
| 320px 해상도 한계 | yolo11n(640px) 모델로 교체 | 중간 (FPS 저하 감수) |
| 파인튜닝 편향 | 강사님께 확인 또는 재학습 | 어려움 |
| take(8) 제한 | take(8) → take(15) | 쉬움 |

**1주일 안에 빠르게 개선하려면:**
1. `confThreshold` 를 0.25~0.30으로 낮춰서 테스트
2. `COCO_KO`에 노트북(63) 추가
3. 결과 비교 후 채택 여부 결정
