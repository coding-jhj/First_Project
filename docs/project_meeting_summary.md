# 프로젝트 회의 내용 정리

## 프로젝트 핵심 방향

### 현재 최우선 목표는 “MVP 완성”
- 모델 성능이나 세부 정확도보다:
  - 서비스가 전체적으로 연결되는지
  - 서버 전송이 되는지
  - 대시보드에서 보이는지
  - 발표/데모 가능한지가 더 중요
- “손이 나오든 팔이 나오든” 같은 디테일보다:
  - MVP 시연 가능 상태가 우선

---

# 현재 구현 상태

## 서버 + DB 연동 상태
현재:
- GCP SQL로 DB 이전 완료
- 서버 연결 MVP 완료
- 탐지 데이터 서버 업로드 가능
- 대시보드에서:
  - 테스트한 핸드폰 종류 확인 가능
  - 탐지 내역 확인 가능

즉:
- “탐지 → 서버 전송 → 저장 → 대시보드 표시” 흐름은 동작 중

---

# 현재 문제점 및 피드백

## “물체 없음” 데이터까지 서버로 전송되는 문제
현재 문제:
- 탐지 실패/빈 결과도 서버로 같이 저장됨

피드백:
- 왜 그런지 로그/디버깅을 통해 직접 원인 분석 필요
- “왜 이런 데이터가 서버에 가는지” 이해하고 수정해야 함
- 단순히 AI 툴만 쓰지 말고:
  - 로직 이해
  - 설계 이해
  - 데이터 흐름 이해가 필요

---

# 서버 저장 데이터 관련 논의

## GPS + 물체 데이터를 왜 저장하는지 목적이 불명확
현재 저장 중:
- GPS
- 탐지 물체 정보

지적된 부분:
- “시각장애인 앞 물체 데이터를 굳이 계속 저장할 필요가 있나?”
- 활용 목적이 불명확함

가능한 활용 예시:
- 특정 위치에서 자주 등장하는 장애물 분석
- 위치 기반 위험 구간 분석

즉:
- 데이터 수집 목적/활용 시나리오를 명확히 정의해야 함

---

# MVP 이후 방향

## 개인 내비게이션 기능 추가 예정
계획 중인 기능:
- 실외 길 안내
- 특정 거리 앞 물체 음성 안내
- 위험 상황 시 진동 알림

하지만:
- 실제 야외 테스트는 시간상 어려움

대안:
- 시뮬레이터 방식 권장
- 가짜 GPS 이동 코드 생성
- 대시보드에서 이동 경로 재현 추천

---

# 발표/포트폴리오 방향

## 지금 중요한 건 “잘 보여주는 것”
강조된 부분:
- 새로운 기능 추가보다:
  - 현재 기능 안정화
  - 시연 흐름 정리
  - 발표 구조 구성
  - 대시보드 시각화
  - 포트폴리오 영상 제작이 중요

특히:
- “뭐가 없어도 잘 보여주는 것도 역량”

---

# 모델/탐지 관련 방향

## 오탐지 문제는 과도하게 신경쓰지 말 것
이유:
- 모바일 환경에서는 원래 발생 가능
- 중요한 건:
  - 시나리오대로 동작하는지
  - 서비스 흐름이 유지되는지

즉:
- 완벽 탐지보다:
  - 실사용 흐름 중심 MVP가 중요

## 트러블슈팅 방식 개선 필요
지적:

# 기술 스택 관련 내용

## 현재 사용 기술
- GCP 서버
- GCP SQL DB
- SQLite 사용 경험 있음
- VSCode Remote SSH 방식 서버 개발 추천 받음

추천된 개발 방식:
- VSCode로 서버 직접 연결
- 서버 환경에서 바로 개발/디버깅
- 이후 Claude 등 AI 도구도 서버 환경에서 활용 가능
