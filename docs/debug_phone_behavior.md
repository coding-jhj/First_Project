# 핸드폰 인식 이상 동작 원인 분석

> 코드 변경사항: confThreshold 0.40 → 0.25 외 없음. 나머지 코드는 그대로임.

---

## 현상 1 — 손에 들면 인식, 테이블에 놓으면 인식 불가

### 원인: 카메라 각도 + 훈련 데이터 편향

YOLO 모델은 이미지를 보고 형태·색상·맥락으로 판단합니다.

**손에 들었을 때:**
```
카메라 → [핸드폰 화면이 크게, 중앙에]
         디스플레이·홈버튼·베젤이 보임
         → 훈련 데이터와 동일한 구도 → 휴대폰으로 인식
```

**테이블에 놓았을 때:**
```
카메라 → [위에서 내려다보는 각도]
         테이블이 화면 대부분 차지
         핸드폰은 테이블 위의 작은 물체
         → 모델이 전체 장면을 "테이블"로 인식
         → 핸드폰은 묻혀버림
```

실제 테스트(cell_phone 이미지 8장)에서 확인:
- 핸드폰을 들거나 정면으로 찍은 이미지 → **휴대폰**으로 탐지
- 테이블 위 핸드폰 이미지 → **테이블**로 탐지

COCO 훈련 데이터 자체가 핸드폰을 주로 "들고 있는" 사진으로 구성돼 있어서,  
위에서 내려다보는 구도는 모델이 학습한 적이 거의 없습니다.

## 현상 2 — 뒤집으면 리모컨으로 인식

### 원인: 시각적 유사성

| | 핸드폰 앞면 | 핸드폰 뒷면 | 리모컨 |
|---|---|---|---|
| 형태 | 직사각형 | 직사각형 | 직사각형 |
| 색상 | 밝은 화면 | 검정/어두운 색 | 검정/어두운 색 |
| 특징 | 홈버튼·베젤 | 카메라 돌출부 | 작은 버튼들 |

핸드폰을 뒤집으면 뒷면(검은 직사각형 + 작은 카메라)이 보이는데,  
이게 리모컨(검은 직사각형 + 작은 버튼들)과 시각적으로 거의 같습니다.

모델은 색상과 형태를 보고 판단하기 때문에 이 둘을 구분하기 어렵습니다.  
코드나 설정으로 해결할 수 없는 모델 한계입니다.

## 현상 3 — everyday_ko에 추가했는데 딱 한 번만 안내

### 원인 A: everyday_ko는 탐지와 무관, 문장 형식만 결정

`everyday_ko`가 실제로 하는 일:

```
SentenceBuilder.kt 65번째 줄

det.classKo in VoicePolicy.everydayKo()
    → "12시 방향에, 핸드폰이 있어요."  ← 이 문장 형식만 결정

everyday_ko에 없으면
    → "12시 방향에, 핸드폰이 있어요."  ← 사실 똑같이 나옴
```

everyday_ko에 핸드폰을 추가해도 **탐지 여부, 탐지 빈도, 안내 빈도**는 전혀 바뀌지 않습니다.

### 원인 B: 5초 쿨다운 (SPEAK_COOLDOWN_MS)

```
MvpPipeline.kt 243번째 줄
private const val SPEAK_COOLDOWN_MS = 5000L
```

같은 물체(같은 트랙 ID)는 **5초에 한 번만** 음성 안내합니다.  
핸드폰을 계속 들고 있어도 처음 한 번 안내하고 5초 뒤에 다시 안내합니다.

### 원인 C: 모델이 프레임마다 다르게 판단

```
1번 프레임: 핸드폰 → 휴대폰으로 탐지 (트랙 생성, 안내)
2번 프레임: 핸드폰 → 테이블로 탐지 (다른 클래스 = 다른 트랙)
3번 프레임: 핸드폰 → 테이블로 탐지
...
12번 프레임 이상 "휴대폰" 트랙 미탐지 → 트랙 삭제
```

```
MvpPipeline.kt 237번째 줄
private const val MAX_MISSED_FRAMES = 12
```

12프레임(약 1초) 이상 탐지 안 되면 트랙이 삭제됩니다.  
핸드폰 트랙이 생겼다 사라졌다를 반복하면서 안내가 끊깁니다.

## 정리

| 현상 | 원인 | 코드/설정으로 해결 가능? |
|---|---|---|
| 테이블 위 인식 불가 | 모델이 위에서 본 핸드폰을 학습 안 함 | ❌ 모델 문제 |
| 뒤집으면 리모컨 | 시각적으로 리모컨과 동일 | ❌ 모델 문제 |
| 한 번만 안내 | 5초 쿨다운 정상 동작 | △ 쿨다운 줄이면 변경 가능 |
| everyday_ko 효과 없음 | 탐지와 무관한 설정 | — (원래부터 탐지에 영향 없음) |

**결론:** 테이블 위 탐지와 뒤집힘 오인식은 yolo11n_320 모델이 이 케이스를 학습하지 않아서 발생하는 문제입니다. 코드나 policy.json 수정으로는 해결이 안 되고, 해당 데이터로 파인튜닝해야 근본 해결됩니다.
