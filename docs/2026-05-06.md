# 2026-05-06 강사 미팅 피드백 및 방향 전환

> 오늘은 코드 수정이 아닌 **설계 방향 전환** 미팅. 아래 내용이 앞으로의 개발 기준이 됨.

---

## 핵심 결론 (한 줄 요약)

**추론은 무조건 디바이스(폰)에서. 서버는 JSON 받아서 뿌리는 것만.**

---

## 1. FPS 문제 — 실시간 서비스의 기준선

- **10fps 미만 = 실시간 서비스 불가**로 판단
- 현재 서버 추론 경로는 6~7fps → 기준 미달
- 서버에서 추론하면 통신 시간(네트워크 왕복)이 추론 시간에 더해져서 구조적으로 FPS가 안 나옴
- **결론: 추론은 폰(온디바이스)이 전담해야 함**

---

## 2. OBB(Oriented Bounding Box) — 사용 보류

- 속도가 기술적으로 실시간에 맞지 않음
- 3D bbox + Depth 조합도 속도 검증이 필요하며 현시점에서 적용 어려움
- **결론: OBB 방향은 일단 접고, n 모델 기반 일반 bbox로 집중**

---

## 3. 거리 추정 방향 변경

- 거리 추정 방식은 박스 크기 기반 외에도 다양한 기법이 있음 (스테레오, 모노큘러 depth 등)
- **이동 중 거리 안내는 하지 않음** — 움직이면 사물이 계속 바뀌어서 물리적으로 안내가 맞지 않음
- **"앞에 뭐가 있어?"처럼 질문했을 때만** 정지 상태 기준으로 안내
- 안내 방식: 최근 1초 / 최근 3초 탐지 결과를 요약해서 전달

---

## 4. 새 아키텍처 설계

```
[Android 폰]
  카메라 → YOLO 추론(온디바이스) → Detection JSON 생성 → 서버 전송

[서버]
  JSON 수신 → DB 저장 또는 직접 처리 → 디바이스별로 결과 뿌리기
  (추론 X, 이미지 처리 X)
```

- 서버가 해야 할 일: JSON 받기 + 디바이스별 결과 배포 + (필요 시) DB 저장
- 서버가 하면 안 되는 일: YOLO 추론, 이미지 분석
- DB 설계: 스키마를 잘 정의해서 탐지 결과 JSON을 DB에 쌓는 구조 고려

---

## 5. 모델 설정

| 항목 | 현재 | 변경 방향 |
|---|---|---|
| 모델 크기 | yolo11s (S) | **yolo11n (N)** — S는 폰에서 너무 무거움 |
| 클래스 수 | COCO 80개 전체 | **필요한 몇 개만** 선택적으로 사용 |
| 정밀도 | (확인 필요) | **float32(32비트)** 명시 |
| 해상도 | (현행) | 낮춰서 속도 확보 검토 |

---

## 6. 음성 안내

- 실외 환경에서 TTS가 이상하게 나오는 문제 존재
- **음성 시나리오를 픽스(고정)** 해야 함 — 케이스별로 어떤 말을 할지 확정
- 보팅 관련 음성 타이밍도 재검토 필요

---

## 7. 보팅(Voting) 관련

- 강사님이 보팅에 대해 질문 → 현재 구조 설명 필요했던 부분
- 현재 보팅은 온디바이스 경로에서만 동작하고 서버 경로에서는 없음 (별도 분석 완료)
- 새 아키텍처(모든 추론을 온디바이스로)로 가면 보팅이 항상 적용되는 구조가 됨

---

## 8. 유사 제품 조사

- 이미 시중에 유사한 앱/서비스가 있을 수 있음
- 있으면 참고해서 구조를 모방하는 것도 좋은 방법
- **조사 필요**: 시각장애인 대상 실시간 장애물 안내 앱 사례

---

## 당장 해야 할 것 (우선순위)

1. **yolo11n 모델로 교체** + 클래스 수 줄이기 + float32 확인
2. **서버에서 추론 코드 제거** → JSON 수신/배포 역할만 남기기
3. **거리 안내 로직 변경** → 질문 시에만, 최근 1~3초 요약으로
4. **음성 시나리오 문서화** → 상황별 발화 내용 픽스
5. **유사 제품 조사** → 참고 레퍼런스 확보
6. **DB 스키마 설계** → 탐지 결과 JSON 저장 구조
