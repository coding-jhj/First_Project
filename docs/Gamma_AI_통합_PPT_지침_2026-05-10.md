# Gamma AI 통합 PPT 제작 지침

작성일: 2026-05-10  
대상 파일: `VoiceGuide (1).pptx`, `VoiceGuide (2).pptx`, `VoiceGuide (3).pptx`  
목표: 세 PPT에서 잘 된 점만 합쳐서, VoiceGuide 프로젝트의 현재 구현 상태에 맞는 최종 발표용 PPT를 만든다.

---

## 1. Gamma AI에게 줄 최종 요청문

아래 내용을 그대로 Gamma AI에 붙여넣어 PPT를 생성한다.

```text
VoiceGuide 프로젝트 최종 발표용 PPT를 만들어줘.

발표 대상은 교수/심사위원이며, 기술 구현을 모르는 사람도 이해할 수 있어야 한다. 하지만 내용은 가볍게 보이면 안 되고, 실제 구현 구조와 검증 결과가 명확히 보여야 한다.

세 기존 PPT의 장점만 합쳐서 만들어줘.

1. VoiceGuide (1).pptx의 장점
- 내용 구성이 가장 완성도 높다.
- 프로젝트 개요, 문제 정의, 고객여정지도, 아키텍처, 파이프라인, 기술 선정 이유, 성능 검증, 데모, 자체 평가까지 발표 흐름이 좋다.
- “서버 추론형 앱이 아니라 Android 온디바이스 즉시 판단 구조”라는 핵심 메시지를 유지한다.

2. VoiceGuide (2).pptx의 장점
- 글자 수가 적고 발표용으로 간결하다.
- 아이보리/화이트 배경, 네이비/딥그린 포인트, 코랄 강조색 조합이 깔끔하다.
- 한 슬라이드에 핵심 메시지 하나만 두는 방식이 좋다.
- 성능 검증 포인트 슬라이드와 자체 평가 슬라이드가 군더더기 없이 좋다.

3. VoiceGuide (3).pptx의 장점
- 표지와 문제 상황 슬라이드에 실제 사용 장면 이미지가 있어 설득력이 좋다.
- 아이콘, 플로우차트, 위험도 패턴 시각화가 직관적이다.
- “길 안내가 아니라 지금 앞의 위험”이라는 메시지가 강하다.
- 생성 결과물 슬라이드가 객체탐지, TTS 문장, JSON 업로드, 대시보드를 한눈에 보여줘 좋다.

위 장점을 합쳐 최종 PPT는 15~16장 내외로 구성해줘. 니즈/문제정의가 약해지지 않도록 초반에는 “국내 규모 → 보행환경 사각지대 → VoiceGuide의 역할” 순서로 설득해줘.

디자인 방향:
- 어두운 보라색/그라데이션 테마는 사용하지 말 것.
- 과한 카드, 둥근 박스, 장식용 도형을 남발하지 말 것.
- 전체 톤은 아이보리/화이트 기반의 깨끗한 발표자료로 만들 것.
- 포인트 컬러는 딥 네이비, 딥 그린, 코랄만 제한적으로 사용할 것.
- 표지와 문제 상황에는 실제 보행/스마트폰 카메라 사용 장면 이미지를 넣을 것.
- 기술 슬라이드는 아이콘, 선형 플로우차트, 표를 사용해 간결하게 보여줄 것.
- 본문은 최소 20pt 이상, 제목은 36pt 이상으로 가독성을 확보할 것.
- 슬라이드 번호, 기본 템플릿 장식, 불필요한 상단 바는 넣지 말 것.
- 발표자가 설명할 수 있도록 슬라이드에는 짧은 키워드만 두고, 긴 문장은 발표자 노트에 넣을 것.
- 초반 니즈/통계 슬라이드와 문제정의 슬라이드 하단에는 출처를 작게 표시할 것.
- 출처가 사용자 제공 수치인 경우 “보건복지부/KOSIS 등록장애인 현황, 2025.12.31 기준(사용자 제공)”처럼 기준일을 명확히 표시할 것.
- 출처 확인이 필요한 수치는 “공식 통계 확인 후 반영”이라고 표시하고 임의로 채우지 말 것.

핵심 메시지:
“VoiceGuide는 스마트폰 카메라 하나로 전방 객체를 감지하고, Android에서 즉시 TTS/진동 안내를 제공하며, 서버는 기록과 대시보드를 담당하는 시각장애인 보행 보조 MVP다.”

반드시 지켜야 할 사실:
- 서버는 YOLO 추론을 하지 않는다.
- Android 앱이 CameraX 프레임을 받아 TFLite YOLO를 온디바이스로 실행한다.
- 서버는 Android가 보낸 JSON을 정규화하고, tracker/DB/SSE 대시보드/이력 조회를 담당한다.
- 현재 주 업로드 경로는 POST /detect다.
- POST /detect_json은 구형 포맷 및 회귀 테스트 호환용으로 남아 있다.
- 거리 추정은 현재 bbox 크기 기반이며, Depth 모델은 향후 개선 방향이다.
- 현재 코드 기준 핵심 추적은 `MvpPipeline.kt`의 IoU 기반 tracking과 EMA smoothing이다. 완전한 ByteTrack 라이브러리 통합처럼 말하지 않는다.
- `yolo26n_float32.tflite`는 fallback 후보/대체 모델로 assets에 포함되어 있으나, “FPS 저하 시 자동 모델 전환이 실기에서 검증 완료”라고 단정하지 않는다. 현재 발표에서는 “fallback 구조/후보를 마련했고, 전환 기준은 실기 FPS 측정으로 확정 예정”이라고 표현한다.
- 서버 `/locations` API는 발표 핵심 기능으로 넣지 않는다. 장소 저장은 Android 내부 저장 중심이며, 서버 장소 API는 발표 범위에서 제외한다.
- 성능 수치를 임의로 만들지 말고, 미측정 항목은 “추가 실기 측정 필요”라고 표시한다.

현재 구현된 핵심 기능:
- Android CameraX 실시간 프레임 입력
- TFLite YOLO 온디바이스 탐지
- yolo11n_320.tflite 기본 모델
- yolo26n_float32.tflite fallback 후보/대체 모델
- raw YOLO 출력 및 end-to-end NMS 출력 지원
- NMS/IoU 기반 중복 제거
- 최근 3프레임 중 2회 이상 등장한 객체만 안내하는 vote 필터
- IoU 0.25 기준 tracking
- EMA smoothing 기반 risk score 안정화
- 위험도 기반 진동 패턴: NONE, SHORT, DOUBLE, URGENT
- 차량 클래스는 risk >= 0.55부터 URGENT로 상향
- SentenceBuilder 기반 로컬 한국어 TTS 문장 생성
- 질문/찾기/들고있는것/장애물 안내 등 음성 모드
- 백그라운드 POST /detect 업로드
- GPS heartbeat POST /gps
- FastAPI 서버, DB 저장, SSE 이벤트, Leaflet 대시보드
- 24시간 탐지 이력 조회 GET /history/{session_id}
- GPS 경로 저장 및 조회 POST /gps/route/save, GET /routes/{session_id}

검증 수치:
- 2026-05-10 기준 자동 테스트: 23 passed, 9 deselected
- 기존 발표자료에 있는 서버 요청 평균 약 26.37ms, NLG 생성 약 0.015ms는 “서버/NLG 검증 수치”로만 사용한다.
- 실제 Android 기기 FPS, 모델별 mAP50, 메모리 사용량, TTS-UI latency, 저조도 성능은 아직 실기 측정 필요 항목으로 분리한다.

PPT 초반 “왜 필요한가”에 넣을 공신력 있는 근거:
- 국내 등록장애인 규모: 2025년 12월 31일 기준 국내 총 등록장애인 수는 2,627,761명, 총인구 대비 약 5.14%다. 이 중 시각장애인은 245,361명으로 전체 등록장애인의 약 9.34%이며, 심한 시각장애인은 44,083명이다. 이 수치는 사용자가 제공한 보건복지부 국내 등록장애인 통계 기준으로 사용한다.
- 공식 비교 근거: 보건복지부 2024년도 등록장애인 현황 통계에서도 등록장애인은 2,631,356명, 전체 인구 대비 5.1%, 시각장애 비중은 9.4%로 확인된다. 따라서 2025년 수치와 큰 흐름이 일치한다.
- 고령화 맥락: 보건복지부 2024년 통계 기준 등록장애인 중 65세 이상 비율은 55.3%다. 시각 보조와 보행 보조는 단순 편의 기능이 아니라 고령화와 함께 중요도가 커지는 안전/이동권 문제다.
- 글로벌 맥락: WHO는 전 세계 최소 22억 명이 근거리 또는 원거리 시각 손상을 갖고 있으며, 시각 손상이 보행 어려움, 낙상 위험, 사회적 고립과 연결될 수 있다고 설명한다.
- 보행환경 위험: 한국소비자원 시각장애인 보행 안전실태조사에서는 수도권 건물 주차장 차량 진·출입로 100곳 중 57곳에 점자블록이 없고, 25곳은 보도가 끊겨 사고 위험이 있는 것으로 조사됐다.
- 공공 보행접근성 문제: 한국시각장애인연합회 시각장애인편의시설지원센터 조사 보도에 따르면 전국 337개 대상시설 주변 점자블록 7,019개 중 적정설치율이 4.0%에 그쳤고, 부적정설치율은 77.3%, 미설치율은 18.7%로 나타났다.

초반 개요용 핵심 문장:
“국내 시각장애인은 약 24.5만 명 규모이며, 보행환경에는 여전히 점자블록 미설치·부적정 설치·차량 진출입로 같은 사각지대가 남아 있습니다. VoiceGuide는 기존 인프라를 대체하는 앱이 아니라, 스마트폰 카메라로 ‘지금 앞의 위험’을 즉시 알려주는 보조 안전 레이어입니다.”

Gamma가 절대 쓰면 안 되는 표현:
- “서버가 영상을 받아 YOLO로 분석한다.”
- “Depth 모델을 적용해 정확한 거리를 계산한다.”
- “ByteTrack 정식 알고리즘을 완전히 통합했다.”
- “YOLO26n으로 FPS 저하 시 자동 전환이 검증 완료됐다.”
- “mAP50, Android FPS, 메모리 사용량이 확보됐다.” 단, 실제 측정값을 사용자가 따로 제공하면 반영 가능.
- “시각장애인 사고를 완전히 예방한다.” 대신 “보행 위험 인지를 보조한다”라고 표현한다.

최종 슬라이드 구성:

1. 표지
제목: VoiceGuide
부제: 실시간 객체 탐지 기반 시각장애인 보행 보조 앱
핵심 문장: 스마트폰 카메라 하나로 전방 객체를 감지하고, 음성·진동으로 즉시 안내합니다.
시각화: 스마트폰 카메라로 보행 장면을 바라보는 이미지.

2. 한 줄 요약
핵심 메시지: 길 안내가 아니라 지금 앞의 위험을 알려주는 앱.
3개 키워드: 온디바이스 추론 / 즉시 음성 안내 / 서버 기록·대시보드

3. 국내 니즈와 문제 규모
내용:
- 2025년 12월 31일 기준 국내 등록장애인 2,627,761명, 총인구 대비 약 5.14%
- 시각장애인 245,361명, 전체 등록장애인의 약 9.34%
- 심한 시각장애인 44,083명
- 보건복지부 2024년 통계에서도 시각장애 비중은 9.4%, 등록장애인 중 65세 이상 비율은 55.3%
강조 문장: 약 24.5만 명의 시각장애인에게 보행 중 “지금 앞의 위험”은 매일 반복되는 안전 문제입니다.

4. 문제 정의
내용:
- 흰 지팡이는 근거리 촉각 정보 중심이다.
- GPS 안내는 경로에는 강하지만 실시간 장애물은 보지 못한다.
- 보행 인프라는 아직 사각지대가 있다. 한국소비자원 조사에서 차량 진·출입로 100곳 중 57곳은 점자블록이 없었고, 25곳은 보도가 끊겨 사고 위험이 있었다.
- 서버 추론형 구조는 네트워크 지연과 오프라인 취약성이 있다.
강조 문장: 보행 중 정말 필요한 정보는 길 안내가 아니라 지금 앞의 위험입니다.

5. 고객여정지도 & Pain Point
상황 3개:
- 횡단보도: 차량·자전거 접근 인지 어려움
- 실내/보도: 의자·문·장애물 미감지
- 혼잡한 장소: 다수 객체 추적과 과다 알림 문제
각 상황별 VoiceGuide 대응을 짧게 넣는다.

6. 해결 방향
3가지 접근:
- Android 온디바이스 추론: 서버 왕복 없이 즉시 판단
- 위험도 기반 안내: 거리·방향·클래스 가중치로 TTS/진동 결정
- 서버 기록 분리: 안내 흐름과 서버 저장/대시보드를 분리

7. 팀 구성 및 역할
4개 파트로 정리:
- Android 앱: CameraX, TFLite, TTS, 진동, UI
- AI 모델: YOLO 모델 변환, 후처리, fallback
- 서버/DB/API: FastAPI, /detect, /gps, status/history/events
- 대시보드/데모: Leaflet, simulator, 데모 시나리오

8. 전체 아키텍처
플로우:
CameraX → TFLite YOLO → vote/dedup → IoU tracking → risk smoothing → TTS/진동
동시에 Android → POST /detect, POST /gps → FastAPI → DB/SSE → Leaflet dashboard
강조:
- Android 즉시 안내
- 서버는 기록과 모니터링
- JSON 기반 단일 데이터 흐름

9. 파이프라인 노드 설명
표 형식:
- CameraX: 프레임 입력
- TFLite YOLO: bbox/class/confidence 산출
- Vote + Dedup: 3프레임 vote, NMS/IoU 중복 제거
- Tracking: IoU 0.25 기준 track 연결
- Risk Analyzer: 거리·중심성·클래스 가중치 + EMA smoothing
- Feedback/Server: TTS·진동·JSON 업로드

10. 기술 선정 이유
내용:
- yolo11n_320: 모바일 기본 탐지 모델
- yolo26n_float32: fallback 후보/대체 모델. 현재 발표에서는 “실기 FPS 측정 후 전환 기준 확정 예정”으로 설명한다.
- TFLite GPU 우선, 실패 시 XNNPACK fallback
- IoU tracking + EMA: 중복 알림과 흔들리는 위험도 완화
주의:
- ByteTrack이라는 표현은 “ByteTrack-lite/IoU 기반 tracking”으로 쓰고, 실제 구현은 IoU tracking 중심이라고 설명한다.

11. 위험도 Smoothing과 진동 패턴
공식:
risk = centerWeight × distanceWeight × classWeight + sizeBoost
패턴:
- SHORT: risk >= 0.35
- DOUBLE: risk >= 0.55
- URGENT: risk >= 0.75
- 차량: risk >= 0.55부터 URGENT 상향
시각화: VoiceGuide (3).pptx의 색상 막대형 위험도 패턴을 참고한다.
```

### 1. 실제 기기 성능 수치

현재 발표자료에 가장 부족한 부분은 Android 실기 성능이다.

채워야 할 표:

| 항목 | 채울 값 | 측정 방법 |
|---|---|---|
| YOLO11n FPS | 실측값 | Android Logcat `VG_PERF` |
| YOLO26n fallback 후보 FPS | 실측값 | 동일 기기에서 해당 모델 실행 |

PPT에는 수치가 없으면 `측정 예정` 또는 `추가 실기 측정 필요`라고 써야 한다.

### 2. 정확도 지표

mAP50은 현재 발표에 넣고 싶지만, 프로젝트 내부에서 검증 데이터셋 기반 수치가 확인되지 않았다. 임의 수치를 만들지 않는다.

채우는 방법:

- 실기 데모용 이미지/영상 프레임을 모아 검증셋을 만든다.
- `tools/verify.py` 또는 별도 검증 스크립트로 모델별 mAP50을 측정한다.
- 측정 전에는 “mAP50: 추가 검증 필요”로 표기한다.

### 3. Depth/거리 추정

현재 구현은 Depth 모델이 아니라 bbox 크기 기반 거리 추정이다. 그러므로 PPT에서 “Depth 적용 완료”라고 쓰면 안 된다.

권장 표현:

- 현재: bbox 크기 기반 거리 추정
- 한계: 단안 카메라라 실제 거리 오차 존재
- 개선: 경량 Depth 모델 또는 거리 캘리브레이션 적용 예정

### 4. ByteTrack 표현

기존 PPT에는 ByteTrack이라는 표현이 강하게 들어가 있지만, 현재 코드 기준 핵심은 `MvpPipeline.kt`의 IoU 기반 tracking과 EMA smoothing이다.

권장 표현:

- `ByteTrack-lite/IoU 기반 tracking`
- 또는 `IoU 기반 간단 tracking`

발표자가 질문받으면 이렇게 답한다:

> 현재 MVP에서는 완전한 ByteTrack 라이브러리 통합보다는, 모바일에서 안정적으로 동작하는 IoU 기반 track 연결과 EMA smoothing을 우선 적용했습니다.

### 5. 서버 역할 정정

서버가 이미지를 받아 YOLO 추론한다고 설명하면 안 된다.

정확한 설명:

- Android: 실시간 탐지, 문장 생성, TTS, 진동
- Server: JSON 정규화, tracker 상태 유지, DB 저장, SSE 대시보드, 이력/경로 조회
- Dashboard: Leaflet 기반 현재 위치, 탐지 객체, 경로, 24시간 이력 시각화

### 6. fallback 표현 정정

기존 PPT에는 “FPS 저하 시 YOLO26n 자동 전환”처럼 보일 수 있는 문장이 있다. 현재 최종 발표에서는 과장하지 않는다.

정확한 표현:

- `yolo11n_320.tflite`를 기본 온디바이스 모델로 사용한다.
- `yolo26n_float32.tflite`는 fallback 후보/대체 모델로 포함되어 있다.
- TFLite 실행 Provider는 GPU FP32를 우선 사용하고, GPU delegate 실패 시 XNNPACK 4 threads로 fallback한다.
- 실제 모델 전환 기준은 Android 실기 FPS 측정 후 확정할 개선 항목이다.

권장 발표 문장:

> 모델 파일과 실행 Provider fallback은 준비되어 있으며, 실제 자동 전환 기준은 실기 FPS 로그를 기반으로 확정할 예정입니다.

### 7. 발표 범위에서 제외할 기능

아래 기능은 코드에 일부 흔적이나 확장 구조가 있어도, 최종 발표 핵심 기능처럼 말하지 않는다.

## 5. 발표자가 기억할 핵심 답변

### 왜 온디바이스인가?

보행 보조는 지연이 생기면 바로 위험해질 수 있다. 그래서 탐지와 안내는 Android에서 즉시 처리하고, 서버는 기록과 모니터링에만 사용한다.

### 왜 서버가 필요한가?

사용자 안내 자체는 앱에서 처리하지만, 탐지 이력, GPS 경로, 현재 상태, 데모 대시보드, 추후 보호자 모니터링을 위해 서버 기록이 필요하다.

### 현재 가장 큰 한계는?

모델 구조보다 실기 측정이 부족한 점이다. 특히 FPS, mAP50, 메모리, TTS latency, 저조도 성능은 발표 전 실제 기기 기준으로 보완해야 한다.

### 이번 MVP에서 가장 잘 된 점은?

탐지, 자연어 안내, 진동, 서버 기록, 대시보드까지 끊기지 않는 하나의 사용 흐름으로 연결한 점이다.
