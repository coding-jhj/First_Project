# 2026-05-04 작업 정리

## 1. 오늘 목표

- 장애물 탐지 성능 개선 방향 결정
  - Android 온디바이스는 빠른 실시간 안내를 유지한다.
  - 서버는 더 정밀한 모델을 사용해 장애물 탐지 정확도를 보강한다.
- FPS 개선 방향 검토
  - 기존 8~9fps 수준에서 10fps 이상을 목표로 한다.
  - 서버 왕복 시간, 업로드 크기, 추론 시간을 줄이는 방향을 확인한다.
- MVP 핵심 기능 테스트 준비
  - 찾기 모드: 핸드폰, 책, 리모컨, 칫솔, 우산
  - 장애물/탐색 모드: 자동차, 사람, 오토바이, 칼, 개
  - 확인 모드와 장애물 모드의 실제 동작 여부 확인
- 서버/Android 문장 생성 정책 차이 확인
  - 서버는 `src/nlg/sentence.py`를 기준으로 문장을 생성한다.
  - Android 온디바이스 fallback은 `SentenceBuilder.kt`가 같은 정책을 따라가야 한다.

## 2. 사용모델

| 위치 | 모델 | 역할 | 상태 |
|---|---|---|---|
| Android 온디바이스 | `yolo11n.onnx` | 네트워크 없이 빠른 1차 장애물 탐지 | 유지 |
| 서버 | `yolo26s.pt` | 서버 정밀 탐지 기본 후보 | 코드 기본값으로 반영 |
| 서버 거리 추정 | Depth Anything V2 | YOLO bbox에 거리/위험도 보강 | 서버 파이프라인 유지 |
| 서버 대체 후보 | `yolo11m.pt`, `yolo26m.pt` | 정확도 비교용 후보 | 추후 실험 |

> 참고: 로컬 코드와 Docker 설정은 서버 기본 모델을 `yolo26s.pt`로 바꿨지만, Cloud Run 운영 서버가 실제로 YOLO26s를 사용하려면 재배포가 필요하다.

## 3. 구동원리

```text
Android 앱 실행
→ CameraX로 프레임 캡처
→ 서버 URL과 네트워크 상태 확인

서버 URL이 없거나 네트워크가 불안정함
→ Android 내부 yolo11n.onnx로 온디바이스 추론
→ SentenceBuilder.kt로 안내 문장 생성
→ TTS/화면/바운딩박스 표시

서버 URL이 있고 네트워크가 정상임
→ 이미지를 /detect API로 업로드
→ 서버가 YOLO26s로 객체 탐지
→ Depth Anything V2 또는 bbox 기반 거리 추정
→ 위험도/방향/문장 생성
→ Android가 응답을 받아 TTS/화면/바운딩박스 표시
```

핵심 구조는 하이브리드 방식이다. Android는 빠른 안전망 역할을 하고, 서버는 더 정밀한 분석 역할을 한다.

## 4. 목표 실제 구동 유,무

| 목표 | 구동 여부 | 확인 내용 |
|---|---|---|
| Android APK 빌드 | 유 | `:app:assembleDebug` 성공 |
| 핸드폰 설치 | 유 | `R5CRA1DK9FM` 기기에 `app-debug.apk` 설치 성공 |
| 앱 실행 | 유 | 프로세스 유지 확인, crash 로그 없음 |
| 온디바이스 경로 | 유 | 화면/로그에서 `경로: ONNX` 확인 |
| 서버 경로 | 유 | 폰 로그에서 `route=server`, `/detect status=200` 확인 |
| 서버 응답 성능 | 유 | 서버 응답이 대략 90~320ms 범위로 확인됨 |
| 운영 서버 YOLO26s 적용 | 부분 | 코드/Docker 설정은 완료, Cloud Run 재배포 필요 |
| 전체 pytest | 무 | 기존 `pytest-cache-files-*` 권한 문제, `conftest` 충돌, `build_hazard_sentence` import 오류로 실패 |

## 5. 코드 수정 전, 후

| 항목 | 수정 전 | 수정 후 |
|---|---|---|
| 서버 YOLO 모델 선택 | `src/vision/detect.py`에 `yolo11n.pt` 하드코딩 | `SERVER_YOLO_MODEL` 환경변수 사용, 기본값 `yolo26s.pt` |
| Docker 모델 다운로드 | 빌드 시 `yolo11n.pt` 사전 다운로드 | 빌드 시 `SERVER_YOLO_MODEL` 사전 다운로드 |
| 환경 변수 예시 | 서버 모델 설정 없음 | `.env.example`에 `SERVER_YOLO_MODEL=yolo26s.pt` 추가 |
| Ultralytics 버전 | `ultralytics==8.3.2` | `ultralytics==8.4.33` |
| 검증 스크립트 기대 버전 | `8.3.2` 기준 | `8.4.33` 기준 |
| Android 빌드 | `tvDetected`, `detectedText` 참조 오류로 실패 | `tvDetected` TextView 복구, `sentence` 표시로 수정 |
| Android 모델 | `yolo11n.onnx` | 변경 없음 |

수정 파일:

- `src/vision/detect.py`
- `Dockerfile`
- `.env.example`
- `requirements.txt`
- `tools/verify.py`
- `android/app/src/main/res/layout/activity_main.xml`
- `android/app/src/main/java/com/voiceguide/MainActivity.kt`

## 6. 내일 목표

- Cloud Run 서버 재배포
  - `SERVER_YOLO_MODEL=yolo26s.pt`가 운영 서버에 실제 반영되는지 확인한다.
  - 배포 후 서버 로그에서 모델 로드 메시지와 `/detect` 성능을 확인한다.
- 서버/Android 문장 정책 동기화
  - `build_hazard_sentence` import 오류를 해결한다.
  - 서버 `sentence.py`와 Android `SentenceBuilder.kt`의 표현 규칙을 맞춘다.
- FPS와 사용성 검증
  - 서버 모드와 ONNX 모드의 FPS, 지연시간, 발열, 배터리 소모를 비교한다.
  - 서버 요청 동시 처리 수와 프레임 전송 간격을 조정한다.
- MVP 시나리오 테스트
  - 찾기: 핸드폰, 책, 리모컨, 칫솔, 우산
  - 탐색/장애물: 자동차, 사람, 오토바이, 칼, 개
  - 확인 모드: 사용자가 묻는 물체를 자연스럽게 설명하는지 확인한다.
- 오탐/누락 기록
  - 계단, 연석, 볼라드, 공사콘, 전동킥보드 등 COCO 기본 클래스에 부족한 장애물 데이터를 따로 수집한다.
  - 이후 YOLO26s 또는 YOLO11 계열 파인튜닝 데이터셋 후보로 정리한다.
