# Meeting Prep 2026-04-29

작성 기준: 강사님 피드백에 맞춰 기능 욕심을 줄이고, 현재 동작하는 MVP를 설명 가능하게 정리합니다.

## 현재 팀 역할

| 이름 | 역할 | 미팅에서 말할 내용 |
|---|---|---|
| 정환주 | 팀장, 서버, 프론트엔드 | MVP 범위, GCP 서버, 전체 통합 상태 |
| 신유득 | Vision, ML | 오탐 클래스, 모델/threshold, Depth fallback |
| 김재현 | Android, UX | FPS 병목, 직렬 처리 개선 계획, 앱 사용 흐름 |
| 임명광 | NLG, 서버 도움 | 응답 문장, 위험도별 안내 문구, 서버 응답 보조 |
| 문수찬 | Voice, Q&A 시트 | STT/TTS 검증, 예상 질문/답변 정리 |

## 강사님께 설명할 핵심

> 현재는 기능 추가보다 검증 가능한 MVP를 우선합니다. Android는 온디바이스 탐지를 유지하고, 서버는 GCP Cloud Run의 `src.api.main:app` 하나로 통일했습니다. 오탐과 FPS 문제는 담당자가 로그와 실험표를 기준으로 개선합니다.

## 실험/연구 노트

| 영역 | 현재 상태 | 다음 기록 |
|---|---|---|
| Vision/ML | COCO 기반 탐지, 오탐 클래스 조정 필요 | 클래스별 오탐 표, threshold 변경 이유 |
| Android | ONNX 온디바이스 동작, FPS 개선 필요 | `VG_PERF` 로그, 직렬/병렬 비교 |
| Server | GCP Cloud Run 기준 | `/health`, `/detect`, `/dashboard` 확인 결과 |
| NLG | 위험도별 문장 생성 | 실제 응답 예시와 수정 이유 |
| Voice/Q&A | Android TTS 우선 | STT/TTS 실패 케이스와 답변 |

## 미팅 전 체크리스트

- [ ] README의 "동작 확인" 항목이 실제 시연 가능한 기능만 담고 있다.
- [ ] GCP 서버 URL로 `/health`가 응답한다.
- [ ] Android 서버 URL을 비워도 온디바이스 탐지가 된다.
- [ ] Android FPS 병목을 로그로 설명할 수 있다.
- [ ] 오탐 사례를 최소 3개 이상 표로 정리했다.
- [ ] 각자 본인 담당 함수를 설명할 수 있다.

## 하지 말 것

| 하지 말 것 | 이유 |
|---|---|
| 새 기능을 완료처럼 말하기 | 검증이 안 되면 질문에서 무너짐 |
| GCP 외 배포를 현재 기준처럼 말하기 | 서버 기준이 흐려짐 |
| 담당자가 아닌 사람이 대신 설명하기 | 역할 분배가 무의미해짐 |
| FPS를 interval 조정만으로 해결했다고 말하기 | 실제 병목을 설명하지 못함 |
