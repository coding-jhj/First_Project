# 경고 피로 억제 — 구현 설계 문서

> 목표: 음성 안내 빈도를 줄이고, 갑작스러운 이벤트는 진동으로 대체한다.  
> 코드를 짜기 전에 **무엇을**, **어디에**, **어떻게** 바꿔야 하는지 이해하기 위한 문서.

---

## 1. 지금 코드가 어떻게 동작하는지

현재 흐름을 한 줄씩 따라가면 이렇다:

```
카메라 프레임
  → YoloDetector.kt       : 물체 탐지 (바운딩박스 + 클래스)
  → MvpPipeline.kt        : 거리 계산 + 위험도 판단 → VibrationPattern 결정
  → MainActivity.kt       : 진동 실행 + TTS 음성 재생
```

그리고 지금 `VibrationPattern`은 4가지다:

| 이름 | 의미 |
|---|---|
| `NONE` | 아무것도 안 함 |
| `SHORT` | 짧은 진동 1번 |
| `DOUBLE` | 진동 2번 |
| `URGENT` | 긴 진동 (위험) |

**문제:** 지금은 진동이 나오면 음성도 같이 나온다. 진동만 나오고 음성은 나오지 않아야 하는 경우가 구분되어 있지 않다.

---

## 2. 무엇을 바꿔야 하는가

### 핵심 아이디어

> "이 이벤트가 **음성까지 필요한가**, 아니면 **진동으로 충분한가**"를 판단하는 단계를 하나 추가한다.

이걸 위해 **이벤트 타입**을 두 가지로 나눈다:

| 타입 | 설명 | 출력 |
|---|---|---|
| `GRADUAL` (점진적) | 사용자가 걸어가면서 천천히 가까워지는 물체 | 음성 + 진동 |
| `SUDDEN` (갑작스러운) | 측면/후방에서 갑자기 나타나거나, 빠르게 접근하는 물체 | **진동만** |

---

## 3. "갑작스럽다"의 기준 — 어떻게 판단하나

### 조건 1: 방향이 측면인 경우

바운딩박스의 중심 X좌표를 기준으로 화면을 3등분한다:

```
화면 가로를 0~1로 정규화했을 때

| 0.0 ~ 0.33 | 0.33 ~ 0.67 | 0.67 ~ 1.0 |
|  좌전방    |    정면     |   우전방   |
```

- **정면(0.33~0.67)** → 음성 + 진동 가능
- **측면(0~0.33 또는 0.67~1.0)** → 진동만

> MvpPipeline.kt에서 바운딩박스 center_x를 이미 계산하고 있으니, 이 값에 조건을 추가하면 된다.

### 조건 2: 접근 속도가 빠른 경우

현재 `tracker.py`에 이미 이런 로직이 있다:

```
이전 프레임 거리 - 현재 프레임 거리 = delta
delta >= 0.8m 이면 "fast approach" 경고
```

이 "fast approach" 상태를 **SUDDEN 이벤트**로 분류하면 된다.

### 조건 3: 이전에 한 번도 탐지되지 않았던 물체가 갑자기 나타남

- 트래킹 ID가 새로 생겼고
- 거리가 이미 2m 이내이면 → SUDDEN

## 4. 어디에 코드를 넣어야 하나

### MvpPipeline.kt에 추가할 것

지금 이 파일이 `VibrationPattern`을 결정하는 곳이다.  
여기에 이벤트 타입 판단 로직을 추가한다.

**지금 구조 (개념):**
```
위험도 계산
  → VibrationPattern 반환
```

**바꿀 구조 (개념):**
```
위험도 계산
  → 방향 판단 (center_x 기준)
  → 접근속도 판단 (delta 기준)
  → EventType 결정 (GRADUAL or SUDDEN)
  → VibrationPattern + shouldSpeak(Boolean) 반환
```

`shouldSpeak`가 `false`이면 MainActivity에서 TTS를 호출하지 않는다.

### MainActivity.kt에 추가할 것

지금 이 파일은 진동 + TTS를 항상 같이 실행한다.  
`shouldSpeak` 값을 받아서 TTS 호출 여부를 결정하도록 수정한다.

**지금 구조 (개념):**
```kotlin
runVibration(pattern)
speakTTS(text)  // 항상 실행
```

**바꿀 구조 (개념):**
```kotlin
runVibration(pattern)
if (shouldSpeak) {
    speakTTS(text)
}
```

## 5. 중복 안내 억제 — 같은 물체를 계속 말하지 않기

이것도 경고 피로의 원인이다.

### 아이디어: "마지막 안내 시각" 기록

각 트래킹 ID마다 "마지막으로 음성 안내한 시각"을 저장해둔다.  
같은 ID에 대해 N초 이내에 다시 음성 안내가 오면 → 무시한다.

```
[추천 억제 간격]
- 일반 물체: 5초
- 긴급 이벤트 (URGENT): 억제 없음 (항상 말함)
```

**어디에 넣나:**  
`MvpPipeline.kt` 안에 `HashMap<Int, Long>`으로 `trackId → lastSpokenTime` 을 관리한다.  
(`System.currentTimeMillis()` 사용)

## 6. 진동 패턴 — 어떤 상황에 어떤 진동을 쓸까

| 상황 | 진동 패턴 | 음성 |
|---|---|---|
| 정면 물체, 10m | 없음 | 없음 |
| 정면 물체, 5m 접근 | SHORT (짧게 1번) | 있음 |
| 정면 물체, 3m 이내 | DOUBLE (2번) | 있음 |
| 정면 물체, 1m 이내 | URGENT (길게) | 있음 |
| 측면 물체 갑자기 등장 | URGENT (길게) | **없음** |
| 빠른 접근 | DOUBLE (2번) | **없음** |

---

## 7. 구현 순서 (권장)

아래 순서로 하나씩 만들고 테스트하면 된다.

```
1단계: MvpPipeline.kt에 EventType 추가
       → GRADUAL / SUDDEN 판단 로직 작성
       → shouldSpeak Boolean 반환 추가

2단계: MainActivity.kt 수정
       → shouldSpeak가 false면 TTS 안 부름

3단계: 중복 억제 추가
       → lastSpokenTime HashMap 추가
       → 5초 이내 재호출 차단

4단계: 실내 고정 물체 테스트
       → 같은 물체에 대해 5초 이상 조용한지 확인
       → 측면에 물체 놓고 진동만 나오는지 확인
```

---

## 8. 주의할 점

- `URGENT` 패턴은 억제 대상에서 **제외**한다. 진짜 위험할 때는 항상 말해야 한다.
- 억제 시간(5초)은 나중에 정책 파일(`policy.json`)에서 조정 가능하도록 상수로 빼두는 게 좋다.
- 진동 패턴은 실제 기기에서 체감이 많이 다르다. 코드 짜고 나서 직접 손에 쥐고 패턴별로 느낌 확인 필수.
