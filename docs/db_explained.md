# DB를 왜 쓰는가 — 쉬운 설명

---

## 1. DB가 뭔가

**DB(데이터베이스) = 데이터를 저장해두는 창고. 엑셀 표랑 비슷하다고 보면 됨.**

폰에서 탐지 결과가 JSON으로 서버에 오는데, 그냥 받아서 화면에 뿌리고 끝내면 **데이터가 사라짐.**  
DB에 저장하면 나중에 꺼내 쓸 수 있음.

---

## 2. 이 프로젝트에서 DB가 필요한 이유

### 시나리오

```
폰(탐지) → JSON → 서버 → DB 저장
                        ↓
                  대시보드/앱에서 조회
```

DB 없이 그냥 흘리면:
- 사용자가 어디서 어떤 장애물을 만났는지 기록이 없음
- 서버를 재시작하면 모든 데이터가 사라짐
- 여러 디바이스의 탐지 결과를 모아볼 수 없음

DB에 저장하면:
- **공간 기억 기능** — "이 장소에 저번에 계단이 있었어" 같은 히스토리 활용 가능
- **디바이스별 관리** — 폰 A / 폰 B 각각의 탐지 결과를 따로 뿌려줄 수 있음
- **나중에 분석 가능** — 어떤 장애물이 자주 탐지됐는지, 어느 위치가 위험한지 등

---

## 3. "스키마를 잘 써야 한다"는 게 무슨 말인가

**스키마 = DB에 데이터를 저장할 때 "어떤 형태로 넣을지" 미리 정해두는 설계도**

엑셀로 비유하면, 컬럼(열) 이름과 타입을 미리 정해두는 것.

### 나쁜 스키마 예시

| id | data |
|---|---|
| 1 | `{"class":"의자","x":0.5,"conf":0.9,"time":"2026-05-06"}` |

→ JSON을 통째로 문자열로 박아버리면 나중에 "의자만 조회"하거나 "특정 시간대만 꺼내기"가 매우 어려움.

### 좋은 스키마 예시

| id | device_id | class_ko | confidence | cx | cy | w | h | detected_at |
|---|---|---|---|---|---|---|---|---|
| 1 | phone-A | 의자 | 0.90 | 0.50 | 0.60 | 0.15 | 0.20 | 2026-05-06 14:32:01 |
| 2 | phone-A | 자동차 | 0.95 | 0.30 | 0.40 | 0.25 | 0.30 | 2026-05-06 14:32:03 |

→ 컬럼이 분리되어 있으면:
- "phone-A의 탐지 결과만" 조회 가능
- "confidence 0.8 이상만" 필터 가능
- "최근 3초 데이터만" 꺼내기 가능

**스키마를 잘 써야 한다 = 나중에 꺼내 쓸 때 편하도록 처음부터 컬럼을 잘 쪼개서 저장하라는 말.**

---

## 4. "웬만해서는 DB에 넣어라"는 게 무슨 말인가

임시로 메모리나 파일에만 갖고 있지 말고, **중요한 데이터는 무조건 DB에 저장해두라는 말.**

강사님이 언급한 맥락:
- 탐지 결과 JSON이 서버에 오면 → 그냥 흘리지 말고 DB에 쌓아라
- 그러면 나중에 "최근 1초", "최근 3초" 요약을 뽑을 때 DB에서 조회하면 됨
- 디바이스별로 다른 데이터를 뿌릴 때도 DB에서 `WHERE device_id = ?`로 꺼내면 됨

---

## 5. 이 프로젝트에서 저장할 것들 (예시 스키마)

### detections 테이블 (탐지 결과)

| 컬럼 | 타입 | 설명 |
|---|---|---|
| id | INTEGER | 자동 증가 기본키 |
| device_id | TEXT | 어느 폰에서 온 데이터인지 |
| session_id | TEXT | 분석 세션 구분 |
| class_ko | TEXT | 탐지된 물체 이름 |
| confidence | REAL | 확신도 (0.0~1.0) |
| cx, cy, w, h | REAL | bbox 좌표 |
| zone | TEXT | 방향 (12시, 3시 등) |
| lat, lng | REAL | GPS 위치 (있을 경우) |
| detected_at | DATETIME | 탐지 시각 |

이렇게 쌓아두면:
```sql
-- 최근 3초 탐지 결과 요약 (질문 응답용)
SELECT class_ko, COUNT(*) as cnt
FROM detections
WHERE device_id = 'phone-A'
  AND detected_at > datetime('now', '-3 seconds')
GROUP BY class_ko
ORDER BY cnt DESC;
```

→ "최근 3초 동안 의자 5번, 사람 3번 탐지됐으니 의자가 가장 가까이 있다" 같은 요약이 가능해짐.
