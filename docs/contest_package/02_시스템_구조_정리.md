# VoiceGuide 시스템 구조 정리

## 한 줄 구조

Android는 즉시 안내를 담당하고, 서버는 기록과 대시보드를 담당합니다.

```text
CameraX → TFLite YOLO → 안정화/위험도 판단 → TTS·진동
                                      ↓
                              POST /detect, /gps
                                      ↓
                              FastAPI → DB/SSE → Dashboard
```

## Android 역할

Android는 사용자가 걷는 순간 필요한 피드백을 처리합니다.

| 단계 | 역할 |
|---|---|
| CameraX | 실시간 프레임 입력 |
| TFLite YOLO | bbox, class, confidence 산출 |
| Vote/Dedup | 순간 오탐과 중복 bbox 완화 |
| IoU Tracking | 같은 객체를 연속 프레임에서 연결 |
| EMA Smoothing | 위치와 위험도 흔들림 완화 |
| Risk 판단 | 거리, 중심성, 클래스 기준으로 위험도 계산 |
| TTS·진동 | 사용자에게 즉시 안내 |
| JSON 업로드 | 서버에 탐지 결과와 위치 기록 전송 |

발표 문장:

> 보행 중 안내는 지연이 작아야 하므로, 객체 탐지와 위험 판단은 Android에서 즉시 처리합니다.

## Server 역할

서버는 모델 서버가 아닙니다. Android가 만든 결과를 저장하고 보여주는 기록·모니터링 서버입니다.

| 엔드포인트 | 역할 |
|---|---|
| `GET /health` | 서버/DB 상태 확인 |
| `GET /api/policy` | Android 정책 설정 배포 |
| `POST /detect` | 현재 주 경로, 탐지 JSON 수신 |
| `POST /detect_json` | 구형 포맷/회귀 테스트 호환 |
| `POST /gps` | 현재 위치 저장 및 SSE 발행 |
| `POST /gps/route/save` | 이동 경로 저장 |
| `GET /routes/{session_id}` | 저장된 경로 목록 조회 |
| `GET /routes/{session_id}/{route_id}` | 특정 경로 조회 |
| `GET /status/{session_id}` | 현재 객체, GPS, track 조회 |
| `GET /events/{session_id}` | SSE 실시간 스트림 |
| `GET /sessions` | 최근 GPS session 목록 |
| `GET /team-locations` | 최근 팀 위치 조회 |
| `GET /history/{session_id}` | 탐지 이력 조회 |
| `GET /dashboard` | Leaflet 대시보드 |
| `POST /spaces/snapshot` | 공간 상태 snapshot 저장 |

발표 문장:

> 서버는 영상을 분석하지 않고, Android가 보낸 탐지 JSON과 GPS 정보를 저장해 대시보드와 이력 조회에 사용합니다.

## DB/대시보드 역할

| 저장 대상 | 용도 |
|---|---|
| 탐지 이벤트 | 언제 어떤 객체가 감지됐는지 기록 |
| 객체 상세 | class, confidence, bbox, distance, risk 저장 |
| GPS 이력 | 이동 경로와 위치 변화 확인 |
| snapshot | session별 최근 공간 상태 복원 |
| recent detections | 구형 포맷 호환 보조 |

## 금지 표현

- 서버가 영상을 받아 YOLO로 분석한다.
- 서버에서 객체 탐지를 수행한다.
- 서버 응답을 기다린 뒤 사용자에게 안내한다.
- 영상 전체를 서버로 업로드한다.
