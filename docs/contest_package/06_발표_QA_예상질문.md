# 발표 Q&A 예상 질문

## Q1. 서버에서 YOLO를 돌리나요?

아닙니다. 실시간 안내는 Android 온디바이스에서 처리합니다. 서버는 Android가 보낸 탐지 JSON과 GPS 정보를 저장하고 대시보드에 보여주는 역할입니다.

## Q2. 왜 서버 추론이 아니라 온디바이스 추론인가요?

보행 중 안내는 지연이 작아야 하기 때문입니다. 네트워크 상황에 따라 안내가 늦어지면 사용자 경험이 떨어질 수 있어, 탐지와 위험 판단은 Android에서 즉시 처리하도록 설계했습니다.

## Q3. 거리 추정은 정확한가요?

현재는 bbox 크기 기반 추정입니다. 정밀 거리 측정에는 한계가 있으므로, 향후 거리 캘리브레이션이나 경량 Depth 모델 검토가 필요합니다.

## Q4. 성능 수치는 어느 정도인가요?

현재 확보된 수치는 자동 테스트 `23 passed, 9 deselected`, 서버 요청 평균 약 `26.37ms`, NLG 생성 약 `0.015ms`입니다. Android FPS, mAP50, 메모리 사용량은 실제 기기 기준 추가 측정 항목으로 분리했습니다.

## Q5. 시각장애인 보행 안전을 완전히 해결하나요?

완전히 해결한다고 표현하지 않습니다. VoiceGuide는 흰 지팡이와 보행 인프라를 대체하는 것이 아니라, 전방 위험 인지를 한 번 더 보조하는 안전 레이어입니다.

## Q6. TTS와 진동은 어떤 역할인가요?

TTS는 탐지 결과를 한국어 안내 문장으로 바꿔 사용자가 화면을 보지 않아도 상황을 이해하도록 돕습니다. 진동은 위험도에 따라 SHORT, DOUBLE, URGENT처럼 다른 패턴으로 즉시 주의를 줍니다.

## Q7. YOLO26n은 현재 자동으로 전환되나요?

자동 전환 검증 완료로 표현하지 않습니다. `yolo26n_float32.tflite`는 fallback 후보/대체 모델로 정리하고, 실제 전환 기준은 기기별 FPS와 정확도 측정 후 확정해야 합니다.

## Q8. 프로젝트의 차별점은 무엇인가요?

길 안내가 아니라 “지금 앞의 위험”에 집중한다는 점입니다. 또한 서버 응답을 기다리지 않고 Android에서 즉시 판단하고, 서버는 기록과 대시보드를 담당하도록 역할을 분리했습니다.

## Q9. 가장 큰 한계는 무엇인가요?

현재는 실제 기기 성능 측정과 저조도 환경 검증이 더 필요합니다. 거리 추정도 bbox 기반이라 정밀한 거리 계산에는 한계가 있습니다.

## Q10. 다음 개발에서 가장 먼저 할 일은 무엇인가요?

Android 실기 로그를 기반으로 FPS, 추론 시간, TTS 지연, 메모리 사용량을 측정하고, 이를 바탕으로 모델 선택과 fallback 기준을 정리하는 것입니다.
