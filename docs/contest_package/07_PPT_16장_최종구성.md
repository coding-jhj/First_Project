# VoiceGuide PPT 16장 최종 구성

## 제작 원칙

- 공모전 제출용 발표자료로 만든다.
- 표지에는 `VoiceGuide`, `3팀`을 정확히 넣는다.
- 각 슬라이드는 핵심 문장 1개와 짧은 키워드 중심으로 구성한다.
- 기술 설명은 표보다 흐름도와 그래픽을 우선한다.
- 서버가 YOLO 추론하는 것처럼 표현하지 않는다.
- 확보된 수치와 추가 측정 필요 항목을 분리한다.
- 제출 필수 목차와 제출 목록을 PPT 안에 빠짐없이 반영한다.

## 제출 조건 반영 방식

PPT 필수 목차는 아래 5개입니다.

| 필수 목차 | PPT 반영 슬라이드 |
|---|---|
| 프로젝트 개요 | 1~3장 |
| 프로젝트 팀 구성 및 역할 | 8장 |
| 프로젝트 수행 절차 및 방법 | 7장, 9~13장 |
| 프로젝트 수행 경과 | 14~15장 |
| 자체 평가 의견 | 16장 |

|---|---|

## 1. 표지

제목: `VoiceGuide`

필수 텍스트:

- `3팀`
- `실시간 객체 탐지 기반 시각장애인 보행 보조 앱`

핵심 문장:

> 스마트폰 카메라 하나로 전방 객체를 감지하고, 음성·진동으로 즉시 안내합니다.

그래픽:

- 실제 YOLO 실험 화면처럼 보이는 스마트폰 카메라 장면
- chair/person/door/stairs bbox 표시
- AI 느낌 사진이나 미래형 이미지 금지

발표자 노트:

> VoiceGuide는 길 안내 앱이 아니라, 스마트폰 카메라로 지금 앞의 위험을 감지하고 음성·진동으로 알려주는 보행 보조 MVP입니다.

- 데모 영상 흐름

발표자 노트:

## 3. 프로젝트 개요

핵심 문장:

> 길 안내가 아니라, 지금 앞의 위험.

본문 키워드:

- 온디바이스 추론
- 즉시 음성·진동
- 서버 기록·대시보드

그래픽:

- 중앙 스마트폰
- 주변 3개 기능 아이콘

발표자 노트:

> 기존 길 안내 앱이 목적지까지의 경로를 돕는다면, VoiceGuide는 이동 중 바로 앞의 객체와 위험을 빠르게 인지하도록 돕습니다.

기능 정리 반영:

- Android 카메라 기반 객체 탐지
- TTS·진동 안내
- 서버 업로드와 대시보드

## 4. 국내 니즈와 문제 규모

핵심 문장:

> 보행 중 전방 위험 인지는 반복되는 실제 문제입니다.

본문:

- 국내 등록장애인: `2,627,761명`
- 시각장애인: `245,361명`
- 심한 시각장애인: `44,083명`

그래픽:

- 큰 숫자 3개 중심 인포그래픽
- 하단 출처 작게 표기

발표자 노트:

> 수치 자체보다 중요한 점은 보행 안전 문제가 충분한 규모와 반복성을 가진다는 점입니다.

---

## 5. 문제 정의

핵심 문장:

> 목적지보다 먼저 필요한 것은 지금 앞의 위험입니다.

본문 키워드:

- 흰 지팡이: 근거리 촉각 중심
- GPS 안내: 경로 중심
- 보행 인프라: 사각지대 존재
- 서버 추론형 앱: 지연과 오프라인 취약성

그래픽:

- 4분할 문제 지도

발표자 노트:

> 흰 지팡이와 GPS는 중요한 도구지만, 스마트폰 카메라로 전방 위험을 미리 감지하고 즉시 알려주는 영역에는 빈틈이 있습니다.

## 6. 고객여정지도와 Pain Point

핵심 문장:

> 상황은 달라도 필요한 것은 즉시 인지 가능한 피드백입니다.

본문:

- 횡단보도: 차량·자전거 접근
- 실내/보도: 의자·문·계단
- 혼잡한 장소: 다수 객체와 과다 알림

그래픽:

- 3단 고객여정지도
- 각 단계에 Pain / VoiceGuide 대응 표시

발표자 노트:

> 이 앱은 특정 장소 하나를 해결하는 앱이 아니라, 이동 중 반복적으로 생기는 전방 위험 인지 문제를 보조합니다.

제출 목록 연결:

- 이 슬라이드는 `고객여정지도(pain point 정리)` 제출 요구를 충족합니다.

## 9. 전체 아키텍처

핵심 문장:

> Android는 즉시 안내, 서버는 기록과 모니터링을 담당합니다.

본문:

- Android: `CameraX → TFLite YOLO → Risk → TTS·진동`
- Server: `POST /detect, /gps → FastAPI → DB/SSE → Dashboard`

그래픽:

- 좌우 분리 아키텍처
- 서버로 영상이 아니라 JSON이 넘어가게 표시

발표자 노트:

> 실시간 판단은 Android에서 끝납니다. 서버는 이미지를 분석하지 않고 Android가 보낸 JSON을 저장·정규화해 대시보드와 이력 조회에 사용합니다.

제출 목록 연결:

- 이 슬라이드는 `전체 파이프라인 구조도(아키텍처)` 제출 요구를 충족합니다.

## 10. Android 파이프라인

핵심 문장:

> 탐지 결과를 안정화한 뒤 안내합니다.

노드:

1. CameraX
2. TFLite YOLO
3. Vote + Dedup
4. IoU Tracking
5. EMA Smoothing
6. Feedback

발표자 노트:

제출 목록 연결:

- 이 슬라이드는 `파이프라인 구성 캡처 및 노드 설명` 제출 요구를 충족합니다.
- 가능하면 실제 앱/코드 기반 파이프라인 캡처 또는 YOLO 실험 화면을 함께 배치합니다.

## 11. 핵심 매칭 알고리즘

핵심 문장:

> 탐지 결과를 객체·방향·거리·위험도에 맞는 안내 시나리오로 바꿉니다.

알고리즘 이름:

```text
객체-위험도 안내 매칭 알고리즘
```

입력:

- Object Class
- Direction
- Distance
- Risk Score

출력:

- TTS 문장
- 진동 패턴
- UI 표시
- 서버 기록

예시:

```text
chair + 정면 + 가까움 + SHORT
→ "정면 약 1.5m에 의자가 있어요."
→ 짧은 진동
```

그래픽:

- 왼쪽: 입력값 카드
- 가운데: Matching Engine
- 오른쪽: TTS/진동/UI/서버 기록 카드

발표자 노트:

> 참고 PDF의 Age, Status, History 매칭 구조처럼 VoiceGuide도 입력값을 조합해 시나리오를 결정합니다. 다만 우리는 사용자 복지 상태가 아니라 객체 종류, 방향, 거리, 위험도를 기준으로 안내를 매칭합니다.

## 12. 기술 선정 이유

핵심 문장:

> 실사용 가능한 속도와 안정성을 위해 온디바이스 구조를 선택했습니다.

본문:

- `yolo11n_320.tflite`: 기본 모델
- `yolo26n_float32.tflite`: fallback 후보
- TFLite GPU/XNNPACK: 모바일 실행
- IoU + EMA: 중복 알림 완화
- 로컬 TTS: 즉시 안내

발표자 노트:

> yolo26n은 fallback 후보로 포함했지만, 자동 전환 기준은 아직 실기 FPS 측정으로 확정해야 할 개선 항목입니다.

## 13. 위험도 계산과 진동 패턴

핵심 문장:

> 모든 객체를 똑같이 알리지 않고 위험도에 따라 다르게 안내합니다.

본문:

- `risk = centerWeight × distanceWeight × classWeight + sizeBoost`
- SHORT: 짧은 1회 진동
- DOUBLE: 짧은 2회 진동
- URGENT: 긴급 진동

그래픽:

- 3단계 위험도 게이지

발표자 노트:

> 위험도는 거리, 중심 위치, 객체 종류를 함께 반영합니다. 차량처럼 위험도가 높은 객체는 더 낮은 기준에서도 강한 진동으로 안내합니다.

## 14. 구현 결과와 서버/API

핵심 문장:

> 탐지 결과는 안내 문장과 서버 기록으로 변환됩니다.

4분할:

- YOLO 탐지 화면
- TTS 문장
- JSON 업로드
- 대시보드

예시:

- `정면 약 1.5m에 의자가 있어요.`
- `POST /detect`
- `POST /gps`

발표자 노트:

> 이 슬라이드는 탐지, 안내, 업로드, 모니터링이 하나의 흐름으로 이어진다는 점을 보여줍니다.

핵심 API:

- `POST /detect`
- `POST /gps`
- `GET /status/{session_id}`
- `GET /events/{session_id}`
- `GET /history/{session_id}`
- `GET /dashboard`

발표자 노트:

> API를 전부 나열하기보다, Android 결과가 서버에 기록되고 대시보드로 확인되는 흐름을 중심으로 설명합니다.

## 15. 수행 경과: 검증 및 데모

핵심 문장:

> 확보된 검증과 추가 측정 항목을 분리했습니다.

확보:

- 자동 테스트 `23 passed, 9 deselected`
- 서버 요청 평균 약 `26.37ms`
- NLG 생성 약 `0.015ms`

추가 측정 필요:

- Android FPS
- mAP50
- 메모리
- TTS-UI latency
- 저조도 성능

데모 흐름:

```text
앱 실행 → bbox 확인 → TTS/진동 → POST 업로드 → 대시보드 확인
```

데모 영상:

- 5~10분 이내 시연영상으로 제출
- 권장 길이는 7분 30초 안팎, 확장 구성은 약 8분 50초
- 영상에는 실제 YOLO 탐지 화면, TTS·진동 안내, 서버 업로드, 대시보드, 검증/시행착오가 들어가야 함

데모 영상 권장 타임라인:

| 구간 | 시간 | 내용 |
|---|---:|---|
| 1 | 0:00~0:30 | 오프닝 |
| 2 | 0:30~1:20 | 문제 정의 |
| 3 | 1:20~2:00 | 서비스 개념 |
| 4 | 2:00~2:50 | 전체 구조 |
| 5 | 2:50~4:20 | 객체 탐지 |
| 6 | 4:20~5:10 | 음성·진동 안내 |
| 7 | 5:10~6:00 | 서버 업로드 |
| 8 | 6:00~6:50 | 대시보드 매칭 |
| 9 | 6:50~7:30 | 모델/검증 |
| 10 | 7:30~8:20 | 시행착오 |
| 11 | 8:20~8:50 | 마무리 |

발표자 노트:

> 서버/NLG 수치는 확보된 값입니다. Android 실기 FPS와 모델 정확도는 실제 기기와 환경에 따라 달라질 수 있어 추가 측정 항목으로 분리했습니다.

필수 목차/제출 목록 연결:

- 이 슬라이드는 `프로젝트 수행 경과`와 `데모 영상(5~10분 이내)` 요구를 충족합니다.

## 16. 자체 평가 의견

핵심 문장:

> 완벽한 탐지 모델보다 실제 보행 중 끊기지 않는 안내 흐름을 우선했습니다.

3열:
