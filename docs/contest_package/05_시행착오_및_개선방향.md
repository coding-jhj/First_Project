# 시행착오 및 개선 방향

## 핵심 관점

시행착오는 실패 기록이 아니라, 프로젝트가 어떤 기준으로 개선되어야 하는지 보여주는 근거입니다.

회의 피드백의 방향은 명확합니다.

```text
새 기능 추가보다 현재 기능 정리.
기능 나열보다 동작 원리 설명.
예쁜 결과물보다 검증 가능한 결과물.
실패 숨기기보다 시행착오와 개선 방향 정리.
```

## 시행착오 표

| 시행착오 | 원인 | 해결/배운 점 | 개선 방향 |
|---|---|---|---|
| 서버 추론 구조 혼선 | Android/Server 역할이 초기에 명확하지 않음 | 즉시 판단은 Android, 기록은 Server로 분리 | 발표와 문서에서 역할 분리 강조 |
| 객체 알림 과다 | 단일 프레임 탐지에 바로 반응하면 오탐/중복 안내 발생 | Vote/Dedup, IoU Tracking, EMA 적용 | 사용자 피로도 기준으로 안내 빈도 조정 |
| 거리 추정 한계 | 현재 bbox 크기 기반 추정 | MVP에서는 위험도 판단 보조 값으로 사용 | 거리 캘리브레이션, 경량 Depth 검토 |
| 성능 수치 부족 | 실제 기기 측정 항목이 아직 충분하지 않음 | 확보된 수치와 미측정 항목 분리 | FPS, mAP50, 메모리, latency 측정 |
| 기능 범위 과다 | 서버, 모델, 음성, 대시보드를 동시에 확장 | MVP 핵심 흐름 중심으로 정리 | 새 기능보다 기존 기능 설명 가능성 우선 |
| 시연 설득력 부족 | 실제 동작 화면보다 설명/이미지 중심이면 약함 | 앱 화면, bbox, 대시보드, 로그를 직접 보여줘야 함 | 시연영상에 실제 화면 중심 구성 |

## 발표용 요약

> 이번 프로젝트에서는 완벽한 탐지 모델보다 실제 보행 중 끊기지 않는 안내 흐름을 우선했습니다. 시행착오는 실패를 숨기는 것이 아니라, 다음 개선 방향을 결정하는 근거로 정리했습니다.

## 향후 개선 방향

| 개선 항목 | 내용 |
|---|---|
| 성능 로그 DB화 | Android FPS, 추론 시간, TTS 지연, 메모리 사용량 기록 |
| 거리 보정 | bbox 기반 거리 추정의 환경별 오차 보정 |
| 모델 비교 | yolo11n/yolo26n 등 모델별 FPS와 정확도 비교 |
| 저조도 검증 | 야간/실내 어두운 환경에서 탐지 안정성 확인 |
| 사용자 테스트 | 실제 접근성 관점에서 안내 문장과 진동 패턴 검증 |
| 대시보드 개선 | 위험 발생 구간, 최근 이력, 경로 분석 강화 |
