## 핵심 원인

### 1. 이미지 상황 문제 (코드·모델 무관)

001, 002, 005, 006번처럼 **테이블 위에 핸드폰이 놓인 사진**은  
테이블이 화면 대부분을 차지해서 두 모델 모두 "테이블"로 탐지합니다.  
이건 모델 한계가 아니라 카메라 각도·구도 문제입니다.

실제 사용 시에도 핸드폰이 작게 잡히면 탐지가 어렵습니다.

### 2. yolo11n_320이 yolo26n보다 일관성이 낮음

- 잘 잡히는 이미지(000, 004, 007): 두 모델 모두 0.7~0.9 수준으로 탐지
- 못 잡는 이미지: yolo26n이 그나마 "휴대폰"으로 잡는 경우가 더 많음
- 003번: yolo11n_320은 "시계(0.37)", yolo26n은 "휴대폰(0.26)"

### 3. confThreshold 0.25 낮춘 효과

현재 0.25로 낮춘 덕분에 003번(0.26)처럼 간신히 잡히는 경우가 살아남습니다.  
0.40으로 올리면 이것도 사라집니다.

## 결론

| 항목 | 내용 |
|---|---|
| yolo26n이 핸드폰 탐지에 더 나음 | 8장 중 4장 성공 vs yolo11n 3장 |
| 두 모델 모두 탁자 위 핸드폰은 못 잡음 | 구도 문제, 코드로 해결 불가 |
| confThreshold 0.25 유지가 맞음 | 낮을수록 경계선 탐지를 살릴 수 있음 |
| "여행가방·기차" 오탐지 | 특정 각도에서 발생, 모델이 핸드폰을 모르는 상태에서 가장 비슷한 것으로 추측 |

## 지금 할 수 있는 것

### 선택 A — yolo26n 우선으로 바꾸기
`TfliteYoloDetector.kt` 37번째 줄 모델 우선순위 변경:
```kotlin
// 현재
listOf("yolo11n_320.tflite", "yolo26n_float32.tflite")

// 변경
listOf("yolo26n_float32.tflite", "yolo11n_320.tflite")
```
핸드폰 탐지는 yolo26n이 더 낫지만, 다른 물체들은 yolo11n_320이 더 나을 수 있음.

### 선택 B — yolo11n_320 그대로 유지 + confThreshold 0.25 유지
현재 상태 유지. 핸드폰은 이미지에 따라 탐지되거나 안 됨.  
"여행가방·기차" 오탐지는 카메라를 핸드폰에 더 가까이 대면 줄어들 수 있음.

### 선택 C — 강사님 파인튜닝 데이터 확인 (근본 해결)
