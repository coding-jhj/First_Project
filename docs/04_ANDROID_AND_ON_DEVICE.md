# Android And On-Device

## Android 앱 역할

Android 앱은 사용자가 실제로 만지는 본 제품입니다. 카메라 프레임을 받고, 온디바이스 ONNX 추론 또는 서버 요청을 통해 음성 안내를 출력합니다.

## 주요 구성

| 파일 | 역할 |
|---|---|
| `MainActivity.kt` | 카메라, 모드, 서버/온디바이스 흐름 제어 |
| `YoloDetector.kt` | ONNX YOLO 추론 |
| `SentenceBuilder.kt` | Android 로컬 안내 문장 생성 |
| `BoundingBoxOverlay.kt` | 탐지 박스 시각화 |
| `VoiceGuideConstants.kt` | 모드, 클래스, 공통 상수 |

## 모드

| 모드 | 설명 |
|---|---|
| 장애물 안내 | 주변 장애물을 탐지해 방향/거리/위험도를 안내 |
| 물건 찾기 | 사용자가 요청한 물체가 보이면 위치 안내 |
| 물건 확인 | 카메라 중앙 또는 가장 가까운 물체를 설명 |

## 온디바이스 우선 구조

Android는 서버 없이도 기본 기능이 유지되도록 설계합니다.

```text
Camera frame
  -> YoloDetector.kt
  -> SentenceBuilder.kt
  -> Android TTS
```

서버가 연결되어 있으면 추가 분석이나 상태 저장을 보조할 수 있지만, 발표 기준 핵심 기능은 온디바이스에서 설명할 수 있어야 합니다.

## 병렬 처리 주의점

카메라 프레임은 계속 들어오지만 ONNX 추론은 무거운 작업입니다. 여러 추론을 동시에 시작한다고 항상 FPS가 좋아지지는 않습니다.

예를 들어 추론 1개가 내부적으로 CPU 2스레드를 쓰는데, 추론 3개를 동시에 돌리면 CPU 경쟁이 생깁니다. 이 경우 각 추론이 더 오래 걸려서 실제 안내가 늦어질 수 있습니다.

따라서 원칙은 다음과 같습니다.

- 오래된 프레임을 모두 분석하지 않습니다.
- 최신 프레임 중심으로 처리합니다.
- in-flight 개수를 제한합니다.
- 프레임 스킵 로그는 실패가 아니라 실시간성을 위한 선택으로 볼 수 있습니다.

예시 로그:

```text
stream frame skipped: route=on_device inFlight=3/3
```

뜻: 이미 온디바이스 분석 3개가 진행 중이라 새 프레임은 버렸다는 의미입니다.

## 문장 규칙 동기화

서버 `src/nlg/sentence.py`와 Android `SentenceBuilder.kt`는 같은 표현 정책을 따라야 합니다.

예:

```text
위험! 바로 앞 자동차! 조심!
바로 코앞에 가방이 있어요. 멈추세요
```

서버와 Android의 문장 차이가 커지면 발표 중 "같은 상황인데 안내가 다르다"는 문제가 생깁니다.
