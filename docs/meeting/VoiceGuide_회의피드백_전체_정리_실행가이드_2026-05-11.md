# VoiceGuide 회의 피드백 전체 정리 실행가이드

이 문서는 `meeting.md` 피드백을 바탕으로, VoiceGuide 프로젝트를 공모전 제출용으로 어떻게 정리해야 하는지 설명한 실행 가이드입니다.

영상 가이드는 이미 따로 만들었으므로, 이 문서는 **영상 외에 반드시 정리해야 할 내용**을 중심으로 다룹니다.

---

## 1. 회의 피드백 핵심 요약

회의 피드백의 핵심은 다음 한 문장입니다.

> 새 기능을 더 붙이는 것보다, 지금까지 만든 기능을 정확히 이해하고 설명 가능한 결과물로 정리해야 한다.

즉, 지금 필요한 것은 기능 추가가 아니라 아래 작업입니다.

- 현재 실제로 동작하는 기능 목록 정리
- 서버, Android, DB, 대시보드 데이터 흐름 정리
- YOLO 모델 입력/출력과 동작 방식 설명
- 모델 선택 이유와 검증 결과 정리
- TTS/STT/자연어 처리 기능의 실제 범위 정리
- 시행착오와 한계 정리
- 발표자료/PPT 구조 정리
- 시연영상과 발표 흐름 연결

## 2. 지금 하면 안 되는 것

공모전 마감 전에는 아래 작업을 우선하지 않는 것이 좋습니다.

- 새로운 기능 추가
- 모델을 갑자기 교체
- 대시보드 대규모 디자인 변경
- 서버 구조 전체 리팩터링
- 검증되지 않은 성능 수치 만들기
- 발표에서 설명하기 어려운 기능을 억지로 넣기

지금은 “더 많이 만들기”보다 “이미 만든 것을 정확히 보여주기”가 중요합니다.

## 4. 기능 정리 문서 작성 방법

먼저 “실제로 되는 기능”과 “아직 개선이 필요한 기능”을 분리해야 합니다.

### 실제 동작 기능

아래 항목은 현재 MVP의 핵심 기능으로 정리합니다.

- Android 카메라 기반 객체 탐지
- TFLite YOLO 온디바이스 추론
- bbox, class, confidence 표시
- 위험도 기반 진동 안내
- 한국어 TTS 안내
- 객체 찾기 모드
- 질문 응답 모드
- 최근 인식 요약
- 서버로 탐지 JSON 업로드
- GPS 업로드
- 대시보드에서 현재 객체, GPS, 탐지 이력 확인

### 개선 필요 기능

아래 항목은 “완료”처럼 쓰지 말고 개선 방향으로 분리합니다.

- Android 실제 기기 FPS 측정
- 모델별 mAP50 측정
- 메모리 사용량 측정
- TTS-UI latency 측정
- 저조도/야간 성능 검증
- 거리 추정 정밀도 개선
- yolo26n fallback 자동 전환 기준 확정
- 실제 사용자 테스트

### 문서에 넣을 표 예시

| 기능 | 현재 상태 | 설명 방식 |
|---|---|---|
| 객체 탐지 | 동작 | Android 온디바이스 TFLite YOLO |
| TTS 안내 | 동작 | 로컬 한국어 안내 문장 생성 |
| 진동 안내 | 동작 | 위험도별 패턴 |
| 서버 저장 | 동작 | POST /detect 중심 |
| 대시보드 | 동작 | 현재 상태와 이력 확인 |
| FPS/mAP50 | 추가 측정 필요 | 임의 수치 작성 금지 |

## 5. 시스템 구조 정리 방법

회의 피드백에서 중요한 부분은 “서버/모델/대시보드/음성 기능이 정리되지 않았다”는 점입니다.

따라서 시스템은 아래처럼 역할을 나누어 설명합니다.

### Android 역할

Android는 사용자의 즉시 안내를 담당합니다.

```text
CameraX
→ TFLite YOLO
→ Vote/Dedup
→ IoU Tracking
→ EMA/Risk
→ TTS·진동
→ 서버 JSON 업로드
```

설명 문장:

> 보행 중 안내는 지연이 작아야 하므로, 객체 탐지와 위험 판단은 Android에서 즉시 처리합니다.

### Server 역할

서버는 영상 추론이 아니라 기록과 모니터링을 담당합니다.

```text
POST /detect
POST /gps
→ FastAPI
→ DB 저장
→ SSE 발행
→ Dashboard 표시
```

설명 문장:

> 서버는 Android가 만든 탐지 JSON과 GPS 정보를 저장하고, 대시보드와 이력 조회에 사용합니다.

### 반드시 피할 표현

- 서버가 영상을 받아 YOLO로 분석한다.
- 서버에서 객체 탐지를 수행한다.
- 서버 응답을 기다린 뒤 사용자에게 안내한다.

## 6. YOLO 모델 설명 정리 방법

모델 설명은 “YOLO 썼다”에서 끝나면 안 됩니다. 입력, 출력, 선택 이유, 한계를 설명해야 합니다.

### 반드시 설명할 항목

| 항목 | 설명 |
|---|---|
| 사용 모델 | yolo11n_320.tflite |
| 실행 위치 | Android 온디바이스 |
| 입력 | CameraX 프레임을 모델 입력 크기에 맞게 변환 |
| 출력 | bbox, class, confidence |
| 후처리 | 중복 제거, vote, tracking, EMA |
| 선택 이유 | 모바일에서 빠른 추론과 즉시 안내가 필요 |
| 한계 | 저조도, 작은 객체, 거리 추정 정확도 |

### 발표용 설명 문장

> YOLO 모델은 카메라 프레임에서 객체의 위치와 종류, 신뢰도를 출력합니다. VoiceGuide는 이 결과를 그대로 말하지 않고, 중복 제거와 tracking, smoothing을 거쳐 사용자에게 필요한 안내로 변환합니다.

### 검증에서 주의할 점

현재 확보되지 않은 수치를 만들어내면 안 됩니다.

- Android FPS: 추가 측정 필요
- mAP50: 추가 측정 필요
- 메모리 사용량: 추가 측정 필요
- 저조도 성능: 추가 측정 필요

## 7. TTS/STT/자연어 기능 정리 방법

회의 피드백에는 자연어/음성 기능 검증도 필요하다는 내용이 있습니다.

현재는 아래처럼 정리하면 됩니다.

### TTS

역할:

- 탐지 결과를 한국어 안내 문장으로 변환
- 사용자가 화면을 보지 않아도 상황을 이해하도록 보조

예시:

```text
정면 약 1.5m에 의자가 있어요.
오른쪽 앞에 장애물이 있습니다.
```

### STT / 음성 명령

역할:

- 사용자가 음성으로 질문하거나 객체 찾기를 요청

예시:

```text
의자 찾아줘
지금 뭐가 있어?
방금 본 것 알려줘
```

### 검증에서 분리할 항목

아래는 아직 정량 수치가 없으면 “추가 측정 필요”로 둡니다.

- STT 인식 정확도
- 음성 명령 성공률
- TTS 시작 지연
- 안내 문장 자연스러움

## 8. 검증 결과 정리 방법

검증 슬라이드와 문서는 반드시 **확보된 수치**와 **추가 측정 필요 항목**을 분리합니다.

### 확보된 수치

| 항목 | 결과 |
|---|---|
| 자동 테스트 | 23 passed, 9 deselected |
| 서버 요청 평균 | 약 26.37ms |
| NLG 생성 시간 | 약 0.015ms |

### 추가 측정 필요

| 항목 | 이유 |
|---|---|
| Android FPS | 실제 기기/환경에 따라 달라짐 |
| mAP50 | 모델별 정량 비교 필요 |
| 메모리 사용량 | 장시간 실행 안정성 확인 필요 |
| TTS-UI latency | 사용자 체감 지연 확인 필요 |
| 저조도 성능 | 실제 보행 환경 검증 필요 |
| STT 인식률 | 음성 명령 정확도 확인 필요 |

### 발표용 문장

> 현재 확보된 정량 수치는 서버 요청과 NLG 생성, 자동 테스트 결과입니다. Android 실기 성능과 모델 정확도는 환경에 따라 달라질 수 있어 추가 측정 항목으로 분리했습니다.

## 9. 시행착오 정리 방법

회의 피드백에서 시행착오는 “실패 기록”이 아니라 “무엇을 배웠는지 보여주는 자료”로 사용하라고 했습니다.

### 시행착오 표 예시

| 시행착오 | 원인 | 해결/배운 점 |
|---|---|---|
| 서버 추론 구조 혼선 | Android/Server 역할 불명확 | 즉시 판단은 Android, 기록은 Server로 분리 |
| 객체 알림 과다 | 단일 프레임 탐지 의존 | Vote/Dedup, Tracking, EMA 적용 |
| 거리 추정 한계 | bbox 크기 기반 추정 | 거리 보정과 Depth 검토 필요 |
| 성능 수치 부족 | 실기 측정 부족 | FPS/mAP50/메모리 측정 계획 분리 |
| 기능 범위 과다 | 서버, 모델, 음성, 대시보드 동시 진행 | MVP 핵심 흐름 중심으로 정리 |

### 발표용 문장

> 시행착오는 실패를 보여주기 위한 것이 아니라, 프로젝트가 어떤 기준으로 개선되어야 하는지 설명하는 근거입니다.

## 13. 예상 질문과 답변 방향

### Q. 서버에서 YOLO를 돌리나요?

```text
아닙니다. 실시간 안내는 Android 온디바이스에서 처리합니다. 서버는 Android가 보낸 탐지 JSON과 GPS 정보를 저장하고 대시보드에 보여주는 역할입니다.
```

### Q. 거리 추정은 정확한가요?

```text
현재는 bbox 크기 기반 추정입니다. 정밀한 거리 측정은 한계가 있어 향후 거리 캘리브레이션이나 경량 Depth 모델 검토가 필요합니다.
```

### Q. 성능 수치는 어느 정도인가요?

```text
현재 확보된 수치는 자동 테스트, 서버 요청 평균, NLG 생성 시간입니다. Android FPS와 mAP50, 메모리는 실제 기기 기준 추가 측정 항목으로 분리했습니다.
```

### Q. 시각장애인 보행 안전을 완전히 해결하나요?

```text
완전히 해결한다고 표현하지 않습니다. VoiceGuide는 흰 지팡이와 보행 인프라를 대체하는 것이 아니라, 전방 위험 인지를 한 번 더 보조하는 안전 레이어입니다.
```

## 14. 최종 체크리스트

### 기능 설명

- [ ] 실제 동작 기능과 개선 필요 기능이 분리됐는가?
- [ ] 기능을 과장하지 않았는가?
- [ ] MVP 범위가 명확한가?

### 기술 설명

- [ ] Android 온디바이스 YOLO 구조가 설명됐는가?
- [ ] 서버 역할이 기록/대시보드로 정리됐는가?
- [ ] 데이터 흐름이 그림으로 설명됐는가?
- [ ] YOLO 입력/출력/후처리가 설명됐는가?
- [ ] TTS/STT 기능 범위가 설명됐는가?

### 검증

- [ ] 확보된 수치와 미측정 항목이 분리됐는가?
- [ ] 없는 FPS/mAP50/메모리 수치를 만들지 않았는가?
- [ ] 시행착오가 개선 근거로 정리됐는가?

### 발표자료

- [ ] PPT에 문제 정의, 목표, 구조, 모델, 검증, 시연, 시행착오, 개선 방향이 모두 들어갔는가?
- [ ] 팀 구성 및 역할이 들어갔는가?
- [ ] 슬라이드가 너무 글로만 되어 있지 않은가?
- [ ] 핵심 문장이 한눈에 보이는가?

### 시연영상

- [ ] 실제 앱 화면이 들어갔는가?
- [ ] YOLO bbox 장면이 보이는가?
- [ ] TTS·진동 안내가 보이는가?
- [ ] 서버 업로드와 대시보드가 보이는가?
- [ ] 모델/검증/시행착오가 들어갔는가?

## 15. 결론

회의 피드백의 방향은 명확합니다.

```text
새 기능 추가보다 현재 기능 정리.
기능 나열보다 동작 원리 설명.
예쁜 결과물보다 검증 가능한 결과물.
실패 숨기기보다 시행착오와 개선 방향 정리.
```

VoiceGuide는 지금부터 아래 4가지만 제대로 정리하면 됩니다.

```text
1. 실제 동작 기능
2. Android-Server 데이터 흐름
3. YOLO/TTS/STT 모델 설명과 검증
4. 시행착오와 향후 개선 방향
```
